## 1. はじめに

### 1.1 この手順書について

本手順書は、開発未経験者がメモ帳アプリケーションを開発できるように、必要なソフトウェアのインストールと設定方法を **1つずつ丁寧に** 説明します。

- **Docker方式（推奨）**: Docker Desktopのみで最短10分で起動可能
- **ローカル開発方式（従来型）**: Java、Maven等を個別にインストールする方式

### 1.2 環境構築で何をするのか

プログラムを書いて動かすには、以下のような「道具」が必要です：

| 道具 | 例え | 実際の役割 | Docker方式 | ローカル方式 |
|------|------|-----------|-----------|------------|
| **Docker** | 仮想環境 | 全ての環境を1つにまとめる | ✅ 必須 | PostgreSQLのみ |
| **JDK** | 大工道具セット | Javaプログラムを書いて実行するための基本ツール | ❌ 不要 | ✅ 必須 |
| **IDE** | 作業台 | プログラムを書きやすくする編集ツール | 🔧 コード編集時のみ | ✅ 必須 |
| **Maven** | 材料調達係 | 必要なライブラリを自動で集めてくれるツール | ❌ 不要 | ✅ 必須 |
| **PostgreSQL** | 倉庫 | データ（メモの内容など）を保存する場所 | ❌ 不要 | 🔧 Dockerで代用可 |
| **Git** | バージョン管理 | プログラムの変更履歴を記録するツール | ✅ 推奨 | ✅ 必須 |

### 1.3 所要時間

**🐳 Docker方式:**
- 初めての方: 約10-20分（Docker Desktopインストール含む）
- 経験者: 約5分

**💻 ローカル開発方式:**
- 初めての方: 約2-3時間
- 経験者: 約30分-1時間

## 2. 環境構築の2つの方法

### 2.1 方法の比較

| 項目 | 🐳 Docker方式 | 💻 ローカル開発方式 |
|------|-------------|-----------------|
| **必要な知識** | Dockerの基本操作のみ | Java、Maven、PostgreSQLの知識 |
| **インストール数** | 1つ（Docker Desktop） | 4-5個（JDK、Maven、PostgreSQL等） |
| **環境の独立性** | ✅ 完全に独立 | ❌ システムに直接インストール |
| **他プロジェクトへの影響** | ✅ なし | ⚠️ バージョン競合の可能性 |
| **初回起動時間** | 5-10分（ビルド含む） | 即座 |
| **ディスク使用量** | 約500MB-1GB | 約200-300MB |
| **デバッグのしやすさ** | 🔧 ログ確認が主 | ✅ IDE統合デバッグ可 |
| **学習目的** | Docker学習に最適 | Java開発の基礎学習に最適 |

### 2.2 推奨する方法

**🐳 Docker方式を推奨する場合:**
- ✅ とにかく最短で動かしたい
- ✅ 他のプロジェクトと環境を分離したい
- ✅ Dockerを学習したい
- ✅ チーム開発で環境を統一したい

**💻 ローカル開発方式を推奨する場合:**
- ✅ Java開発の基礎から学びたい
- ✅ IDE統合デバッグを活用したい
- ✅ 既にJava環境が整っている
- ✅ Dockerをインストールできない環境

**初学者へのアドバイス:**
まずDocker方式で動かしてアプリケーションの動作を理解し、その後ローカル開発方式でJava開発の詳細を学ぶのがおすすめです。

## 3. 🐳 Docker方式（推奨・最短）

### 3.1 必要なもの

- **Docker Desktop** のみ

### 3.2 インストール手順

#### Windows

1. **Docker Desktop のダウンロード**
   - [Docker Desktop for Windows](https://www.docker.com/products/docker-desktop/) にアクセス
   - "Download for Windows" をクリック

2. **インストール**
   - ダウンロードした `Docker Desktop Installer.exe` を実行
   - インストールウィザードに従って進める
   - WSL 2 のインストールを求められた場合は指示に従う

3. **Docker Desktop の起動**
   - インストール後、Docker Desktop を起動
   - 初回起動時はチュートリアルが表示されるがスキップ可能

4. **インストール確認**
   ```powershell
   docker --version
   docker compose version
   ```

#### macOS

1. **Docker Desktop のダウンロード**
   - [Docker Desktop for Mac](https://www.docker.com/products/docker-desktop/) にアクセス
   - Appleシリコン（M1/M2）版またはIntel版を選択してダウンロード

2. **インストール**
   - ダウンロードした `.dmg` ファイルを開く
   - Docker.app を Applications フォルダにドラッグ

3. **Docker Desktop の起動**
   - Applications から Docker を起動
   - メニューバーに Docker アイコンが表示されるまで待つ

4. **インストール確認**
   ```bash
   docker --version
   docker compose version
   ```

### 3.3 アプリケーションの起動

1. **プロジェクトのクローン**
   ```bash
   git clone https://github.com/fukt-dev/learn-memo-app-traditional.git
   cd learn-memo-app-traditional
   ```

2. **全サービスの起動**
   ```bash
   docker compose up --build
   ```

   初回起動時の流れ：
   - Mavenが依存ライブラリをダウンロード（2-3分）
   - Javaアプリケーションをビルド（2-3分）
   - PostgreSQLコンテナを起動
   - アプリケーションコンテナを起動
   - **起動時にFlywayがデータベースマイグレーションを自動実行**（`memos` テーブルが作成される）

3. **起動確認**
   ブラウザで http://localhost:8080/memos にアクセス

   ログに以下が表示されていれば成功です:
   ```
   Successfully applied 1 migration to schema "public"
   Started MemoAppApplication in X.XXX seconds
   ```

4. **停止**
   ```bash
   # Ctrl+C で停止
   # または別ターミナルで
   docker compose down
   ```

### 3.4 日常的な使い方

```bash
# バックグラウンド起動
docker compose up -d

# ログ確認
docker compose logs -f app

# 停止
docker compose down

# データベースも含めて完全リセット
docker compose down -v
```

**これで完了！** 次は[セクション6: プロジェクトのセットアップ](#6-プロジェクトのセットアップ)へ進んでください。

### 4.2 Java Development Kit (JDK)

#### 4.2.1 JDKとは何か

**JDK (Java Development Kit)** は、Javaプログラムを書いて実行するための基本ツールセットです。

**何ができるようになるか**:
- Javaのソースコード（`.java`ファイル）を書ける
- Javaプログラムをコンパイル（翻訳）して実行できる形（`.class`ファイル）にする
- 作成したプログラムを実行する

**なぜJDK 17が必要か**:
- Spring Boot 3.x は Java 17以上が必要
- 最新の機能とセキュリティアップデートを利用できる

#### 4.2.2 JDKのインストール

##### Windows

**方法1: Oracle JDK（推奨）**

1. **ダウンロード**
   - [Oracle JDK ダウンロードページ](https://www.oracle.com/java/technologies/downloads/) にアクセス
   - "Java 17" を選択
   - "Windows" タブから "x64 Installer" をダウンロード

2. **インストール**
   - ダウンロードした `.exe` ファイルをダブルクリック
   - インストーラーの指示に従う
   - インストール先は変更せず、デフォルトのまま進む（例: `C:\Program Files\Java\jdk-17`）

3. **環境変数の設定**

   環境変数とは、Windowsがプログラムを見つけるための「住所録」のようなものです。

   **手順**:
   ```
   1. スタートメニューを右クリック → 「システム」をクリック
   2. 「システムの詳細設定」をクリック
   3. 「環境変数」ボタンをクリック
   4. 「システム環境変数」の「新規」をクリック

   変数名: JAVA_HOME
   変数値: C:\Program Files\Java\jdk-17
   （実際のインストール先に合わせて調整）

   5. 「システム環境変数」から「Path」を選択して「編集」
   6. 「新規」をクリックして以下を追加:
      %JAVA_HOME%\bin

   7. すべてのウィンドウで「OK」をクリック
   ```

**方法2: Adoptium（無料版、推奨代替）**

1. [Adoptium](https://adoptium.net/) にアクセス
2. "Temurin 17 (LTS)" を選択
3. Windows用のインストーラーをダウンロード
4. インストーラーを実行（環境変数は自動設定されます）

##### macOS

**Homebrew使用（推奨）**

```bash
# Homebrewがインストールされていない場合
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# JDK 17をインストール
brew install openjdk@17

# シンボリックリンク作成
sudo ln -sfn /opt/homebrew/opt/openjdk@17/libexec/openjdk.jdk /Library/Java/JavaVirtualMachines/openjdk-17.jdk

# 環境変数設定（~/.zshrc または ~/.bash_profile に追加）
echo 'export JAVA_HOME=/Library/Java/JavaVirtualMachines/openjdk-17.jdk/Contents/Home' >> ~/.zshrc
echo 'export PATH="$JAVA_HOME/bin:$PATH"' >> ~/.zshrc

# 設定を反映
source ~/.zshrc
```

##### Linux (Ubuntu/Debian)

```bash
# パッケージリストを更新
sudo apt update

# JDK 17をインストール
sudo apt install openjdk-17-jdk -y

# 環境変数設定（~/.bashrc に追加）
echo 'export JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64' >> ~/.bashrc
echo 'export PATH="$JAVA_HOME/bin:$PATH"' >> ~/.bashrc

# 設定を反映
source ~/.bashrc
```

#### 4.2.3 インストール確認

ターミナル（Windows: コマンドプロンプトまたはPowerShell、macOS/Linux: ターミナル）を開いて以下を実行：

```bash
# Javaのバージョン確認
java -version

# 期待される出力例:
# openjdk version "17.0.9" 2023-10-17
# OpenJDK Runtime Environment (build 17.0.9+9)
# OpenJDK 64-Bit Server VM (build 17.0.9+9, mixed mode, sharing)

# 環境変数の確認
echo $JAVA_HOME  # macOS/Linux
echo %JAVA_HOME%  # Windows (コマンドプロンプト)
$env:JAVA_HOME    # Windows (PowerShell)
```

**重要**: バージョンが `17.x.x` 以上であることを確認してください。

#### ✅ JDKインストール完了チェックリスト

- [ ] `java -version` コマンドでバージョン17以上が表示される
- [ ] `JAVA_HOME` 環境変数が正しく設定されている
- [ ] 新しいターミナル/コマンドプロンプトで上記のコマンドが動作する

**トラブル時の確認ポイント：**
- バージョンが表示されない → PATH環境変数の設定を確認
- 古いバージョンが表示される → 複数のJDKがインストールされていないか確認
- コマンドが見つからない → JDKのインストールをやり直す

### 4.3 統合開発環境 (IDE)

#### 4.3.1 IDEとは何か

**IDE (Integrated Development Environment)** は、プログラムを書くための「超高機能なエディタ」です。

**IDEがあると便利なこと**:
- ✅ コードを途中まで書くと、続きを自動で提案してくれる
- ✅ エラーがあると赤線で教えてくれる
- ✅ クラスやメソッドをクリック1つで定義場所に飛べる
- ✅ デバッグ（バグ探し）が簡単にできる

**IDEは次の2つからどちらか1つを選べばOK**（両方入れる必要はありません）:

- **IntelliJ IDEA Community Edition**（4.3.2で説明）
- **Eclipse (Pleiades All in One)**（4.3.3で説明）── 日本語化済み・JDK / Maven / Lombok 設定済みのオールインワン配布版。**これを選ぶと [4.2 JDK](#42-java-development-kit-jdk) と [4.4 Maven](#44-ビルドツール-maven) のインストールが丸ごと不要になります**

迷ったら、個別インストールの手間が少ないEclipse (Pleiades All in One) がおすすめです。

#### 4.3.2 IntelliJ IDEA のインストール

**IntelliJ IDEA** は、Java開発で最も人気のあるIDEです。

**すべてのOS共通手順（Community Edition = 無料版）**:

1. **ダウンロード**
   - [IntelliJ IDEA ダウンロードページ](https://www.jetbrains.com/idea/download/) にアクセス
   - **Community Edition（無料）** の「Download」をクリック
   - OSに合わせたインストーラーをダウンロード

2. **インストール**
   - Windows: `.exe` ファイルを実行
   - macOS: `.dmg` ファイルを開いてアプリケーションフォルダにドラッグ
   - Linux: `.tar.gz` を展開して `bin/idea.sh` を実行

3. **初回起動設定**
   ```
   1. IntelliJ IDEAを起動
   2. 「Do not import settings」を選択（初回のみ）
   3. テーマを選択（DarculaまたはLight）
   4. プラグインのインストールは後でOK
   5. 「Start using IntelliJ IDEA」をクリック
   ```

#### 4.3.3 Eclipse (Pleiades All in One) のインストール

**Eclipse (Pleiades All in One)** は、Eclipse本体に日本語化・JDK・Maven・Lombok等をあらかじめ組み込んだ「オールインワン」配布版です。MergeDoc Projectという有志コミュニティが配布しています。

**ダウンロード手順**:

1. 検索エンジンで「Pleiades」を検索し、MergeDoc Project のサイトを開く
2. 「Eclipse 2026」→ エディション「Java」→「Windows x64」の「JRE あり (Full Edition)」を選択
3. ファイル例: `pleiades-2026-06-java-win-64bit-jre_20260705.exe`（約1.3GB）
4. **⚠️ 罠**: ダウンロードページには大きな「Download Now」ボタン風の**広告**が表示されます。本物のダウンロードリンクは、ページ中にある `ftp.jaist.ac.jp`（JAISTミラー）へのテキストリンクで、通常はクリックすると自動でダウンロードが始まります
5. **ファイルの正当性確認（推奨）**: ダウンロードしたexeファイルが改ざんされていないか確認できます。ページに記載されたMD5値と、以下のコマンドの結果が一致することを確認してください
   ```powershell
   Get-FileHash "ダウンロードしたファイル" -Algorithm MD5
   ```

**インストール（展開）手順**:

1. ダウンロードしたexeを実行する（自己解凍形式であり、インストーラーではないため、レジストリや環境変数は一切変更されません）
2. **Microsoft Defender SmartScreenの「Windows によって PC が保護されました」という警告が出たら「詳細情報」→「実行」をクリック**（コード署名のない自己解凍形式のファイルであるための警告です。MD5照合済みであれば、安全に続行できます）
3. 展開先を指定する。**`C:\pleiades` などのCドライブ直下の浅いパス**にすること（デフォルトの推奨設定に従えばOK）
   - ❌ `Program Files` 配下 → 権限問題でプラグイン追加や自己更新が壊れる
   - ❌ 深いフォルダ・日本語を含むパス → Windowsの260文字パス制限に引っかかり、展開や起動に失敗する
4. 展開が終わったら、以下を実行して起動する:
   ```
   C:\pleiades\2026-06\eclipse\eclipse.exe
   ```

**Pleiadesに含まれるもの（＝個別インストールが不要になるもの）**:

| 同梱物 | 内容 | 対応する手順書セクション |
|--------|------|--------------------------|
| JDK | 8 / 11 / 17 / 21 / 25 の5バージョンが同梱・IDEに登録済み（Eclipse本体はJDK 21で動作） | [4.2 JDK](#42-java-development-kit-jdk) が不要になる |
| Maven | m2e（EclipseのMaven統合）を内蔵 | [4.4 Maven](#44-ビルドツール-maven) が不要になる |
| Lombok | `eclipse.ini` に `-javaagent:lombok.jar` 設定済み | 素のEclipseでは手動設定が必要な箇所 |
| 文字コード | `-Dfile.encoding=UTF-8` 設定済み。コンソールの日本語ログが文字化けしない | ― |
| 日本語化 | メニュー・設定画面が日本語 | ― |

**フォント設定（推奨）**:

デフォルトのエディタ・コンソールフォントは「ＭＳ ゴシック 9pt」で、現代の高解像度モニタでは小さく読みづらいです。以下の手順で変更できます。

1. メニュー「ウィンドウ」→「設定」→「一般」→「外観」→「色とフォント」→「基本」→「テキスト・フォント」→「編集」
2. おすすめのフォント: **BIZ UDゴシック 11pt**（Windows 11標準搭載・日本語込みで読みやすい）、または **Consolas 11pt**（日本語は自動的に別フォントで代替表示される）
3. 「テキスト・フォント」を変更すると、Javaエディタとコンソールのフォントが一括で変わります

#### ✅ Eclipseインストール完了チェックリスト

- [ ] eclipse.exeが起動できる
- [ ] メニューが日本語で表示されている
- [ ] (推奨)テキスト・フォントを変更した

#### 4.3.4 IDEの基本設定（IntelliJ IDEAの場合）

**JDKの設定確認**:
```
1. 「Configure」→「Project Structure」を開く
2. 「Project」タブを選択
3. 「Project SDK」でJDK 17が選択されているか確認
4. なければ「Add SDK」→「JDK」から先ほどインストールしたJDK 17を選択
```

**Maven設定確認（自動検出されているはず）**:
```
1. 「Configure」→「Settings」（macOS: Preferences）
2. 「Build, Execution, Deployment」→「Build Tools」→「Maven」
3. Maven home directoryが設定されているか確認
   （IntelliJ IDEAにバンドルされているMavenが使われます）
```

#### ✅ IDEインストール完了チェックリスト

- [ ] IntelliJ IDEAが起動できる
- [ ] 日本語化プラグイン（Japanese Language Pack）がインストールされている（任意）
- [ ] Mavenプロジェクトをインポートできる（確認は後ほど）

**トラブル時の確認ポイント：**
- 起動しない → システム要件を確認（メモリ不足の可能性）
- プラグインが見つからない → インターネット接続を確認

### 4.6 Docker

#### 4.6.1 Dockerとは何か

**Docker（ドッカー）** は、アプリケーションを「コンテナ」という箱に入れて動かす技術です。

**コンテナとは**:
- アプリケーションとその動作に必要なものをすべて詰め込んだ「パッケージ」
- どの環境（Windows、macOS、Linux）でも同じように動く

**例え話**:
- **従来**: 引っ越しで家具をバラバラに運ぶ（大変！）
- **Docker**: すべてをコンテナ（箱）に詰めて運ぶ（簡単！）

**PostgreSQLをDockerで動かすメリット**:
- ✅ インストールが簡単（1コマンドで起動）
- ✅ 不要になったら簡単に削除できる
- ✅ 複数バージョンを同時に動かせる
- ✅ 本番環境と同じ環境を作れる

#### 4.6.2 Dockerのインストール

インストール手順は [3.2 インストール手順](#32-インストール手順) と同じです（Docker方式・ローカル方式共通）。

Linux (Ubuntu) の場合:

```bash
# 古いバージョンを削除（存在する場合）
sudo apt-get remove docker docker-engine docker.io containerd runc

# 必要なパッケージをインストール
sudo apt-get update
sudo apt-get install ca-certificates curl gnupg lsb-release

# Dockerの公式GPGキーを追加
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# リポジトリ追加
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Docker Engineをインストール
sudo apt-get update
sudo apt-get install docker-ce docker-ce-cli containerd.io docker-compose-plugin

# 現在のユーザーをdockerグループに追加（sudo不要にする）
sudo usermod -aG docker $USER

# ログアウト・ログインして設定を反映
```

#### 4.6.3 Dockerインストール確認

```bash
# Dockerバージョン確認
docker --version

# Docker Composeバージョン確認
docker compose version

# 動作確認（Hello Worldコンテナを実行）
docker run hello-world

# 期待される出力:
# Hello from Docker!
# This message shows that your installation appears to be working correctly.
```

#### 4.6.4 PostgreSQLコンテナの起動

**本プロジェクトでは、リポジトリ同梱の `docker-compose.yml` を使うのが基本です**（[6. プロジェクトのセットアップ](#6-プロジェクトのセットアップ) 参照）。

Dockerの理解のために、単発コマンドでも試せます:

```bash
# PostgreSQLコンテナを起動（お試し用。プロジェクトでは compose を使う）
docker run --name postgres-trial \
  -e POSTGRES_USER=memoapp_user \
  -e POSTGRES_PASSWORD=memoapp_pass \
  -e POSTGRES_DB=memoapp \
  -p 5432:5432 \
  -d postgres:16-alpine

# 起動確認
docker ps
```

**コマンドの説明**:
- `--name postgres-trial`: コンテナに名前を付ける
- `-e POSTGRES_USER=...`: 環境変数を設定（データベースユーザー名）
- `-e POSTGRES_PASSWORD=...`: データベースパスワード
- `-e POSTGRES_DB=memoapp`: 最初に作成するデータベース名
- `-p 5432:5432`: ポート番号をマッピング（ホスト:コンテナ）
- `-d`: バックグラウンドで実行
- `postgres:16-alpine`: 使用するDockerイメージ（PostgreSQL 16 軽量版）

**お試しコンテナの後片付け**（プロジェクトのcomposeとポートが衝突するため、試したら削除してください）:
```bash
docker stop postgres-trial
docker rm postgres-trial
```

#### ✅ Dockerインストール完了チェックリスト

- [ ] `docker --version` コマンドでバージョンが表示される
- [ ] `docker compose version` コマンドでバージョンが表示される（Docker Compose V2）
- [ ] Docker Desktopが起動している（Windows/macOS）
- [ ] `docker run hello-world` コマンドが成功する

**トラブル時の確認ポイント：**
- コマンドが見つからない → Dockerのインストールをやり直す
- Docker Desktopが起動しない → システム要件を確認（WSL2が必要な場合あり）
- 権限エラー（Linux） → 現在のユーザーをdockerグループに追加

# JAVA_HOMEが設定されているか確認

**解決方法**: JDK 17をインストールして、JAVA_HOMEを更新

### 7.2 Maven関連

#### 問題: `mvn: command not found`

**原因**: Mavenがインストールされていない、またはPATHが通っていない

**解決方法**: IntelliJ IDEA内蔵のMavenを使う（IntelliJ IDEAのターミナルから実行）

#### 問題: 依存関係のダウンロードに失敗

**解決方法**:
```bash
# キャッシュをクリアして再実行
mvn clean install -U
```

### 7.3 Docker関連

#### 問題: `Cannot connect to the Docker daemon`

**原因**: Dockerが起動していない

**解決方法**:
- Windows/macOS: Docker Desktopを起動
- Linux: `sudo systemctl start docker`

#### 問題: ポート5432が既に使用されている

**原因**: 別のPostgreSQLインスタンスが既に動いている

**解決方法**:
```bash
# 既存のPostgreSQLを停止
sudo systemctl stop postgresql  # Linux
brew services stop postgresql   # macOS

```
※ モダン版（learn-memo-app-modern）のPostgreSQLコンテナはホスト側5433を使います。両方同時に動かす場合はポートが重ならないようにしてください。

#### 問題: `FATAL: password authentication failed`

**原因**: パスワードが間違っている

**解決方法**: `application.yml` のパスワードと docker-compose.yml のパスワードを確認

#### 問題: `connection refused`

**原因**: PostgreSQLコンテナが起動していない

**解決方法**:
```bash
# コンテナが起動しているか確認
docker compose ps

# 起動していない場合
docker compose up -d postgres
```

### 7.5 アプリケーション起動関連

#### 問題: `Port 8080 is already in use`

**原因**: 別のアプリケーションがポート8080を使用している

**解決方法**:
```bash
# 使用中のプロセスを確認
# Windows
netstat -ano | findstr :8080

# macOS/Linux
lsof -i :8080

# プロセスを停止するか、別のポートを使う（application.ymlを編集）
server:
  port: 8081
```

**補足（Docker方式でアプリを動かしたことがある場合）**: `memoapp-spring` コンテナが8080を使用中のことがあります → `docker stop memoapp-spring` で停止してからローカル起動してください（`docker start memoapp-spring` で元に戻せます）。

#### 問題: `Flyway migration failed` / `Migration checksum mismatch`

**原因**:
- checksum mismatch: 適用済みのマイグレーションファイルが書き換えられた（Flywayはファイル内容のチェックサムを記録しており、コメント変更でも不一致になります）
- その他: マイグレーションSQLの文法エラーなど

**解決方法**:
```bash
# 学習環境ではデータベースのリセットが最も確実（データは消えます）
docker compose down -v
docker compose up -d postgres
mvn spring-boot:run
```

**補足**: `mvn flyway:repair` のようなコマンドは flyway-maven-plugin を導入しているプロジェクト専用で、**本プロジェクトでは使えません**。マイグレーションの状態は `flyway_schema_history` テーブルをSQLで直接確認できます:
```bash
docker compose exec postgres psql -U memoapp_user -d memoapp -c "SELECT version, description, checksum, success FROM flyway_schema_history;"
```
