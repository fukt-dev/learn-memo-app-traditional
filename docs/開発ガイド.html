2.2 各ディレクトリの役割

2.2.1 src/main/java - Javaソースコード

ディレクトリ役割例

開発の流れ

3.1 新機能開発の基本ステップ

新しい機能(例: メモにカテゴリーを追加)を開発する場合の流れ:

ステップ1: データベース設計 → Flywayマイグレーションファイルを「追加」（V2__xxx.sql。既存のV1は変更しない） ステップ2: データアクセス層 → Entity修正 → Mapperインターフェース修正 → Mapper XML（2waySQL）修正 ステップ3: ビジネスロジック層 → DTO修正 → Service修正 ステップ4: プレゼンテーション層 → Controller修正 → Thymeleafテンプレート修正 ステップ5: テスト → 単体テスト作成（mvn test で回帰確認） → 実機で動作確認

3.2 具体例: 「メモの新規作成」機能を層ごとに追う

実装済みの新規作成機能を例に、各層で何をしているかを実際のコードで追います。

3.2.1 データベース定義(db/migration/V1__Create_memos_table.sql より抜粋)

CREATE TABLE memos (
    id BIGSERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
ポイント:日時はDBの DEFAULT とトリガーが自動設定するため、アプリ側からは送りません。

3.2.2 Entityクラス(entity/Memo.java の構造)

@Data
@NoArgsConstructor
@AllArgsConstructor
public class Memo {
    private Long id;                  // id 列に対応
    private String title;             // title 列に対応
    private String content;           // content 列に対応
    private LocalDateTime createdAt;  // created_at 列に対応（自動変換）
    private LocalDateTime updatedAt;  // updated_at 列に対応（自動変換）
}
ポイント:created_at → createdAt の変換は map-underscore-to-camel-case: true の設定で自動化されています。

3.2.3 Mapperインターフェース(mapper/MemoMapper.java より抜粋)

@Mapper
public interface MemoMapper {
    void insert(Memo memo);   // 実際のSQLは MemoMapper.xml に記述
}

3.2.4 Mapper XML(mybatis/mapper/MemoMapper.xml より抜粋)

<insert id="insert" useGeneratedKeys="true" keyProperty="id">
    INSERT INTO memos (title, content) VALUES (#{title}, #{content})
</insert>
ポイント:useGeneratedKeys="true" により、DB が採番した ID が memo.getId() で取れるようになります。

3.2.5 DTO(dto/MemoDto.java より抜粋)

@Data
@NoArgsConstructor
@AllArgsConstructor
public class MemoDto {

    private Long id;

    @NotBlank(message = "タイトルを入力してください")
    @Size(min = 1, max = 200, message = "タイトルは1文字以上200文字以内で入力してください")
    private String title;

    @NotBlank(message = "本文を入力してください")
    private String content;

    // 表示用（フォーマット済み文字列。フォームからは受け取らない）
    private String createdAtFormatted;
    private String updatedAtFormatted;
}
ポイント:バリデーションはDTOに集約。日時は LocalDateTime ではなく表示用のフォーマット済み文字列で持ちます(変換は fromEntity() / toEntity() が担当)。

3.2.6 Service(service/MemoService.java より抜粋)

@Service
@Slf4j
@RequiredArgsConstructor
public class MemoService {

    private final MemoMapper memoMapper;

    @Transactional
    public void create(MemoDto dto) {
        Memo memo = dto.toEntity();     // DTO → Entity 変換
        memoMapper.insert(memo);        // DB登録（IDが採番される）
        log.info("メモを登録しました: id={}, title={}", memo.getId(), memo.getTitle());
    }
}
ポイント:このアプリの Service は具象クラス1つです(MemoService インターフェース + MemoServiceImpl という分離はしていません。理由は設計書 §9 参照)。

3.2.7 Controller(controller/MemoController.java より抜粋)

@Controller
@Slf4j
@RequiredArgsConstructor
@RequestMapping("/memos")
public class MemoController {

    private final MemoService memoService;

    @GetMapping("/new")
    public String newMemo(Model model) {
        model.addAttribute("memoDto", new MemoDto());
        return "memos/new";
    }

    @PostMapping("")
    public String create(
            @Valid @ModelAttribute MemoDto memoDto,
            BindingResult bindingResult,
            RedirectAttributes redirectAttributes
    ) {
        if (bindingResult.hasErrors()) {
            return "memos/new";     // エラー時は入力画面に戻る（入力値は保持される）
        }
        memoService.create(memoDto);
        redirectAttributes.addFlashAttribute("successMessage", "メモを登録しました");
        return "redirect:/memos";   // PRGパターン（二重送信防止）
    }
}

第 4 章

3層アーキテクチャの実践

4.1 層ごとの責務

4.1.1 プレゼンテーション層(Controller)

責務:

  • ✅ HTTPリクエストを受け取る
  • ✅ リクエストパラメータをDTOに変換
  • ✅ バリデーションエラーの確認
  • ✅ Serviceを呼び出す
  • ✅ Viewに渡すデータを準備

やってはいけないこと:

  • ❌ ビジネスロジックを書く(Serviceの仕事)
  • ❌ SQLを実行する(Mapperの仕事)
  • ❌ 直接Entityを扱う(DTOを使う)

良い例(実装より) — 一覧・検索の分岐すらServiceに任せる:

@GetMapping("")
public String list(@RequestParam(name = "keyword", required = false) String keyword, Model model) {
    // 「キーワードが空なら全件」の判断は search() の中にある。Controllerは分岐しない
    List<MemoDto> memos = memoService.search(keyword);
    model.addAttribute("memos", memos);
    model.addAttribute("keyword", keyword);
    return "memos/list";
}

悪い例 — ControllerがSQLを実行している:

@PostMapping
public String create(@Valid MemoDto memoDto) {
    // ❌ ControllerでSQLを実行している（層の責務違反）
    String sql = "INSERT INTO memos (title, content) VALUES (?, ?)";
    jdbcTemplate.update(sql, memoDto.getTitle(), memoDto.getContent());
    return "redirect:/memos";
}

4.1.2 ビジネスロジック層(Service)

責務:

  • ✅ ビジネスルールの実装
  • ✅ トランザクション管理(@Transactional。参照系は readOnly = true)
  • ✅ DTOとEntityの相互変換
  • ✅ 複数のMapperを組み合わせた処理

やってはいけないこと:

  • ❌ HTTPリクエスト/レスポンスを扱う
  • ❌ 直接SQLを書く

実装例(MemoService.search() より) — ビジネスルール「空なら全件」をServiceが持つ:

@Transactional(readOnly = true)
public List<MemoDto> search(String keyword) {
    if (keyword == null || keyword.trim().isEmpty()) {
        return findAll();   // 空キーワードは全件取得にフォールバック
    }
    List<Memo> memos = memoMapper.searchByKeyword(keyword);
    return memos.stream().map(MemoDto::fromEntity).collect(Collectors.toList());
}

4.1.3 データアクセス層(Mapper)

責務:

  • ✅ データベースへのCRUD操作
  • ✅ SQLの実行
  • ✅ 結果をEntityにマッピング

やってはいけないこと:

  • ❌ ビジネスロジックを実装
  • ❌ トランザクション制御(Serviceの仕事)

実装(MemoMapper.java のシグネチャ):

@Mapper
public interface MemoMapper {
    List<Memo> findAll();
    Optional<Memo> findById(Long id);       // 「無いかもしれない」を型で表現
    void insert(Memo memo);
    int update(Memo memo);                  // 戻り値 = 更新件数
    int deleteById(Long id);
    List<Memo> searchByKeyword(@Param("keyword") String keyword);
    // 以下はページング実装演習用（現在未使用）
    List<Memo> findAllWithPaging(@Param("limit") int limit, @Param("offset") int offset);
    long count();
}

DTOとEntity・コーディング規約

4.2 DTOとEntityの使い分け

4.2.1 Entity(エンティティ)

用途: データベーステーブルの1行を表現

特徴:

  • データベースのカラムと1対1で対応
  • バリデーションアノテーションは付けない
  • MyBatisで取得したデータを格納

4.2.2 DTO(データ転送オブジェクト)

用途: 画面とサービス層の間でデータ転送

特徴:

  • バリデーションアノテーションを付ける
  • 画面に必要なフィールドのみ含む(Entityと完全に一致しなくてもOK)
  • このアプリでは日時をフォーマット済み文字列(createdAtFormatted)で持ち、テンプレート側のフォーマット処理を不要にしている

4.2.3 変換方法(実装より)

変換ロジックは MemoDto 自身が持ちます(Serviceに変換コードが散らばらない):

// Entity → DTO（日時フォーマット込み）
MemoDto dto = MemoDto.fromEntity(memo);

// DTO → Entity（id・title・contentのみコピー。日時はDBが管理するためコピーしない）
Memo memo = dto.toEntity();

6.1 ログ出力

6.1.1 ログレベル(実装での使い分け)

log.debug("メモを取得します: id={}", id);       // 開発時のデバッグ情報

7.2 Serviceテスト(MemoServiceTest より抜粋)

@ExtendWith(MockitoExtension.class)     // JUnit 5 に Mockito を組み込む
class MemoServiceTest {

    @Mock                               // MemoMapper のモック（偽物）を生成
    private MemoMapper memoMapper;

    @InjectMocks                        // モックを注入したテスト対象を生成
    private MemoService memoService;

    @Test
    @DisplayName("存在しないIDなら、MemoNotFoundException がスローされる")
    void throwsException_whenMemoDoesNotExist() {
        // Arrange: 「DBに該当行がなかった」状況を再現
        when(memoMapper.findById(999L)).thenReturn(Optional.empty());

        // Act & Assert: 例外がスローされることを検証
        assertThatThrownBy(() -> memoService.findById(999L))
                .isInstanceOf(MemoNotFoundException.class)
                .hasMessageContaining("id=999");
    }
}

ポイント:

  • @SpringBootTest は使いません。Service単体のロジック検証にSpringコンテナは不要で、Mockitoだけの方が圧倒的に速い
  • 異常系(例外)も仕様の一部としてテストする

7.3 Controllerテスト(MemoControllerTest より抜粋)

@WebMvcTest(MemoController.class)   // Web層だけを起動するスライステスト
class MemoControllerTest {

    @Autowired
    private MockMvc mockMvc;        // 擬似HTTPリクエストを送る道具

    @MockitoBean                    // DIコンテナ内のServiceをモックに差し替え
    private MemoService memoService;

    @Test
    @DisplayName("存在しないIDなら 404 + エラー画面が返る")
    void returns404_whenMemoNotFound() throws Exception {
        when(memoService.findById(999L))
                .thenThrow(new MemoNotFoundException("メモが見つかりません: id=999"));

        mockMvc.perform(get("/memos/999"))
                .andExpect(status().isNotFound())
                .andExpect(view().name("error/error"));
    }
}
ポイント(落とし穴):Spring Boot 3.4以降、@MockBean は非推奨になり @MockitoBean(org.springframework.test.context.bean.override.mockito パッケージ)に置き換えられました。古い記事のコードをコピーすると非推奨警告が出ます。

7.5 テストのベストプラクティス

テスト名の付け方:

  • @DisplayName で日本語の説明を付ける(テスト結果が仕様書のように読める)
  • @Nested でメソッドごとにグループ化する

テストの構造(AAA パターン):

  1. Arrange(準備): テストに必要なデータやモックの動作を準備
  2. Act(実行): テスト対象のメソッドを実行
  3. Assert(検証): 期待する結果と実際の結果を比較

回帰テストの考え方:
バグを直したら、その再発を防ぐテストを残す。このリポジトリでは「存在しないIDで200が返っていたバグ」「編集エラー時に日時欄が空になるバグ」の回帰テストが MemoControllerTest にあります。

カバレッジの目安:

  • Serviceクラス: 80%以上(ビジネスロジックの主要部分)
  • Controller: 主要な画面遷移とバリデーション、エラーパス
  • Mapper: 複雑なSQLのみ(基本的なCRUDは省略可)

7.6 Mapperテスト(学習課題)

Mapperテストは実際のDBが必要です。本プロジェクトのDDLはPostgreSQL専用構文(BIGSERIAL・トリガー等)のため、H2での代替はできません。

実務で一般的な構成は @MybatisTest + Testcontainers(テスト実行時に使い捨てのPostgreSQLコンテナを自動起動するライブラリ)です。挑戦する場合は「Testcontainers PostgreSQL JUnit5」で調べてみてください。

よくある問題と解決方法

8.1 コンパイルエラー

問題: Cannot resolve symbol (Lombok関連)

原因: Lombok プラグインがインストールされていない

解決方法:

1. IntelliJ IDEA → Settings → Plugins 2. "Lombok" を検索してインストール 3. 再起動

問題: Cannot resolve method 'xxx'

原因: Mavenの依存関係がダウンロードされていない

解決方法:

IntelliJ IDEA の右サイドバー → Maven → Reload All Maven Projects

8.2 実行時エラー

問題: Port 8080 is already in use

# application.yml
server:
  port: 8081  # 別のポートに変更

問題: Connection refused (PostgreSQL)

docker compose up -d postgres
docker compose ps  # 起動確認

問題: Flyway migration failed / Migration checksum mismatch

原因: 適用済みマイグレーションファイルの変更(コメント変更でもchecksumが変わる)、またはSQL文法エラー

# 学習環境ではDBリセットが最も確実（データは消えます）
docker compose down -v
docker compose up -d postgres
mvn spring-boot:run

※ mvn flyway:repair 等のMavenコマンドは flyway-maven-plugin を導入していない本プロジェクトでは使えません。マイグレーションはアプリ起動時に自動実行されます。

8.3 画面表示エラー

問題: Thymeleafで null が表示される / エラーになる

原因: Controllerでモデルに値を設定していない

@GetMapping("")
public String list(Model model) {
    model.addAttribute("memos", memoService.search(null));  // ← これを忘れずに！
    return "memos/list";
}

問題: CSSが反映されない

ブラウザで Ctrl+F5 (強制リロード)

問題: エラー画面は出るのにHTTPステータスが200になる

原因: @ExceptionHandler メソッドに @ResponseStatus を付けていない(例外クラス側の指定はハンドラ捕捉時には効かない)

解決方法: GlobalExceptionHandler の各ハンドラのように、メソッド側に @ResponseStatus を明示する
