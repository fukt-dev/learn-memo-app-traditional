## 1. はじめに

### 1.1 このガイドについて

本ガイドは、メモ帳アプリケーションを開発する際の **実践的な手順** を説明します。

**前提知識**:
- 環境構築が完了していること（[環境構築手順.md](./環境構築手順.md) 参照）
- 設計書を一読していること（[設計書.md](./設計書.md) 参照）

### 1.2 学習の進め方

```
1. コードを読む → 2. 写経する → 3. 動かす → 4. 改造する
```

**推奨学習フロー**:
1. **まず動かす**: アプリを起動して動作を確認
2. **コードを読む**: 各行が何をしているか理解する（ソースの学習コメントが「なぜ」を説明しています）
3. **小さく変える**: メッセージや数値を変えてみる
4. **新機能を追加**: 理解が深まったら実装演習（ページング等）に挑戦

## 2. プロジェクト構造の理解

### 2.1 全体像

```
learn-memo-app-traditional/
├── src/                          ← ソースコードがここにある
│   ├── main/                     ← アプリケーション本体
│   │   ├── java/                 ← Javaコード
│   │   │   └── com/example/memoapp/
│   │   │       ├── controller/   ← 【プレゼンテーション層】画面リクエストを処理
│   │   │       ├── service/      ← 【ビジネスロジック層】業務ルールを実装
│   │   │       ├── mapper/       ← 【データアクセス層】DBアクセス
│   │   │       ├── entity/       ← データベーステーブルに対応するクラス
│   │   │       ├── dto/          ← 画面とサービス間でデータをやり取りするクラス
│   │   │       └── exception/    ← エラー処理（例外→HTTPステータス変換）
│   │   └── resources/            ← 設定ファイルとテンプレート
│   │       ├── application.yml   ← アプリケーション設定
│   │       ├── templates/        ← Thymeleaf HTML テンプレート
│   │       │   └── fragments/    ← 共通ヘッダー・フッター（th:fragment）
│   │       ├── static/           ← CSS、JavaScript
│   │       ├── mybatis/mapper/   ← MyBatis SQL定義（2waySQL）
│   │       └── db/migration/     ← Flyway マイグレーションSQL
│   └── test/                     ← テストコード（Service/Controller/DTO の30テスト）
├── docs/                         ← ドキュメント
├── pom.xml                       ← Maven設定ファイル（依存関係定義）
├── docker-compose.yml            ← Docker設定（PostgreSQL + アプリ起動）
└── README.md                     ← プロジェクト概要
```

### 2.2 各ディレクトリの役割

#### 2.2.1 `src/main/java` - Javaソースコード

| ディレクトリ | 役割 | 例 |
|-------------|------|---|
| `controller/` | **HTTPリクエストを受け取る** | ボタンをクリック → Controllerが反応 |
| `service/` | **ビジネスロジックを実装** | 「キーワードが空なら全件取得」の判断 |
| `mapper/` | **データベースにアクセス** | SELECT, INSERT, UPDATE, DELETE |
| `entity/` | **データベースの1行を表現** | memosテーブルの1行 = Memoオブジェクト |
| `dto/` | **画面とサービス間でデータ転送** | 入力フォームのデータを受け取る + バリデーション |
| `exception/` | **エラーを処理** | メモが見つからない → 404エラー画面 |

#### 2.2.2 `src/main/resources` - リソースファイル

| ファイル/ディレクトリ | 役割 | 変更頻度 |
|---------------------|------|---------|
| `application.yml` | アプリケーション設定（DB接続情報など） | 少ない |
| `templates/` | Thymeleaf HTMLテンプレート | 多い |
| `static/css/` | スタイルシート（デザイン） | 中程度 |
| `static/js/` | JavaScript（動的な動作） | 中程度 |
| `mybatis/mapper/` | SQLファイル（2waySQL） | 中程度 |
| `db/migration/` | データベース変更スクリプト | 少ない（**追加のみ**。適用済みファイルの変更はchecksumエラーになる） |

### 2.3 データの流れ

```
【ブラウザ】
    ↓ HTTPリクエスト（例: GET /memos）
【Controller】
    ↓ Serviceを呼び出す
【Service】
    ↓ Mapperを呼び出す
【Mapper】
    ↓ SQLを実行
【PostgreSQL】
    ↑ データを返す
【Mapper】
    ↑ Entityオブジェクトに変換
【Service】
    ↑ DTOに変換
【Controller】
    ↑ Viewにデータを渡す
【Thymeleaf】
    ↑ HTMLを生成
【ブラウザ】
    画面表示！
```

## 3. 開発の流れ

### 3.1 新機能開発の基本ステップ

新しい機能（例: メモにカテゴリーを追加）を開発する場合の流れ：

```
ステップ1: データベース設計
  → Flywayマイグレーションファイルを「追加」（V2__xxx.sql。既存のV1は変更しない）

ステップ2: データアクセス層
  → Entity修正
  → Mapperインターフェース修正
  → Mapper XML（2waySQL）修正

ステップ3: ビジネスロジック層
  → DTO修正
  → Service修正

ステップ4: プレゼンテーション層
  → Controller修正
  → Thymeleafテンプレート修正

ステップ5: テスト
  → 単体テスト作成（mvn test で回帰確認）
  → 実機で動作確認
```

### 3.2 具体例: 「メモの新規作成」機能を層ごとに追う

実装済みの新規作成機能を例に、各層で何をしているかを**実際のコード**で追います。

#### 3.2.1 データベース定義（`db/migration/V1__Create_memos_table.sql` より抜粋）

```sql
CREATE TABLE memos (
    id BIGSERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

ポイント: 日時はDBの `DEFAULT` とトリガーが自動設定するため、アプリ側からは送りません。

#### 3.2.2 Entityクラス（`entity/Memo.java` の構造）

```java
@Data
@NoArgsConstructor
@AllArgsConstructor
public class Memo {
    private Long id;                  // id 列に対応
    private String title;             // title 列に対応
    private String content;           // content 列に対応
    private LocalDateTime createdAt;  // created_at 列に対応（自動変換）
    private LocalDateTime updatedAt;  // updated_at 列に対応（自動変換）
}
```

ポイント: `created_at` → `createdAt` の変換は `map-underscore-to-camel-case: true` の設定で自動化されています。

#### 3.2.3 Mapperインターフェース（`mapper/MemoMapper.java` より抜粋）

```java
@Mapper
public interface MemoMapper {
    void insert(Memo memo);   // 実際のSQLは MemoMapper.xml に記述
}
```

#### 3.2.4 Mapper XML（`mybatis/mapper/MemoMapper.xml` より抜粋）

```xml
<insert id="insert" useGeneratedKeys="true" keyProperty="id">
    INSERT INTO memos (title, content) VALUES (#{title}, #{content})
</insert>
```

ポイント: `useGeneratedKeys="true"` により、DB が採番した ID が `memo.getId()` で取れるようになります。

#### 3.2.5 DTO（`dto/MemoDto.java` より抜粋）

```java
@Data
@NoArgsConstructor
@AllArgsConstructor
public class MemoDto {

    private Long id;

    @NotBlank(message = "タイトルを入力してください")
    @Size(min = 1, max = 200, message = "タイトルは1文字以上200文字以内で入力してください")
    private String title;

    @NotBlank(message = "本文を入力してください")
    private String content;

    // 表示用（フォーマット済み文字列。フォームからは受け取らない）
    private String createdAtFormatted;
    private String updatedAtFormatted;
}
```

ポイント: バリデーションはDTOに集約。日時は `LocalDateTime` ではなく**表示用のフォーマット済み文字列**で持ちます（変換は `fromEntity()` / `toEntity()` が担当）。

#### 3.2.6 Service（`service/MemoService.java` より抜粋）

```java
@Service
@Slf4j
@RequiredArgsConstructor
public class MemoService {

    private final MemoMapper memoMapper;

    @Transactional
    public void create(MemoDto dto) {
        Memo memo = dto.toEntity();     // DTO → Entity 変換
        memoMapper.insert(memo);        // DB登録（IDが採番される）
        log.info("メモを登録しました: id={}, title={}", memo.getId(), memo.getTitle());
    }
}
```

ポイント: このアプリの Service は**具象クラス1つ**です（`MemoService` インターフェース + `MemoServiceImpl` という分離はしていません。理由は設計書 §9 参照）。

#### 3.2.7 Controller（`controller/MemoController.java` より抜粋）

```java
@Controller
@Slf4j
@RequiredArgsConstructor
@RequestMapping("/memos")
public class MemoController {

    private final MemoService memoService;

    @GetMapping("/new")
    public String newMemo(Model model) {
        model.addAttribute("memoDto", new MemoDto());
        return "memos/new";
    }

    @PostMapping("")
    public String create(
            @Valid @ModelAttribute MemoDto memoDto,
            BindingResult bindingResult,
            RedirectAttributes redirectAttributes
    ) {
        if (bindingResult.hasErrors()) {
            return "memos/new";     // エラー時は入力画面に戻る（入力値は保持される）
        }
        memoService.create(memoDto);
        redirectAttributes.addFlashAttribute("successMessage", "メモを登録しました");
        return "redirect:/memos";   // PRGパターン（二重送信防止）
    }
}
```

#### 3.2.8 Thymeleafテンプレート（`templates/memos/new.html` より抜粋）

```html
<form th:action="@{/memos}" th:object="${memoDto}" method="post" class="memo-form">
    <div class="form-group">
        <label for="title">タイトル</label>
        <input type="text"
               th:field="*{title}"
               class="form-control"
               th:classappend="${#fields.hasErrors('title')} ? 'is-invalid' : ''"
               placeholder="タイトルを入力してください">
        <span th:if="${#fields.hasErrors('title')}"
              th:errors="*{title}"
              class="error-message">エラーメッセージ</span>
    </div>
    <!-- 本文も同様。ボタン等は実ファイル参照 -->
</form>
```

ポイント: `th:field="*{title}"` が `id` / `name` / `value` を自動生成します。ヘッダー・フッターは `th:replace="~{fragments/parts :: siteHeader('new')}"` で共通部品を取り込んでいます。

## 4. 3層アーキテクチャの実践

### 4.1 層ごとの責務

#### 4.1.1 プレゼンテーション層（Controller）

**責務**:
- ✅ HTTPリクエストを受け取る
- ✅ リクエストパラメータをDTOに変換
- ✅ バリデーションエラーの確認
- ✅ Serviceを呼び出す
- ✅ Viewに渡すデータを準備

**やってはいけないこと**:
- ❌ ビジネスロジックを書く（Serviceの仕事）
- ❌ SQLを実行する（Mapperの仕事）
- ❌ 直接Entityを扱う（DTOを使う）

**良い例（実装より）** — 一覧・検索の分岐すらServiceに任せる:
```java
@GetMapping("")
public String list(@RequestParam(name = "keyword", required = false) String keyword, Model model) {
    // 「キーワードが空なら全件」の判断は search() の中にある。Controllerは分岐しない
    List<MemoDto> memos = memoService.search(keyword);
    model.addAttribute("memos", memos);
    model.addAttribute("keyword", keyword);
    return "memos/list";
}
```

**悪い例** — ControllerがSQLを実行している:
```java
@PostMapping
public String create(@Valid MemoDto memoDto) {
    // ❌ ControllerでSQLを実行している（層の責務違反）
    String sql = "INSERT INTO memos (title, content) VALUES (?, ?)";
    jdbcTemplate.update(sql, memoDto.getTitle(), memoDto.getContent());
    return "redirect:/memos";
}
```

#### 4.1.2 ビジネスロジック層（Service）

**責務**:
- ✅ ビジネスルールの実装
- ✅ トランザクション管理（@Transactional。参照系は readOnly = true）
- ✅ DTOとEntityの相互変換
- ✅ 複数のMapperを組み合わせた処理

**やってはいけないこと**:
- ❌ HTTPリクエスト/レスポンスを扱う
- ❌ 直接SQLを書く

**実装例（`MemoService.search()` より）** — ビジネスルール「空なら全件」をServiceが持つ:
```java
@Transactional(readOnly = true)
public List<MemoDto> search(String keyword) {
    if (keyword == null || keyword.trim().isEmpty()) {
        return findAll();   // 空キーワードは全件取得にフォールバック
    }
    List<Memo> memos = memoMapper.searchByKeyword(keyword);
    return memos.stream().map(MemoDto::fromEntity).collect(Collectors.toList());
}
```

#### 4.1.3 データアクセス層（Mapper）

**責務**:
- ✅ データベースへのCRUD操作
- ✅ SQLの実行
- ✅ 結果をEntityにマッピング

**やってはいけないこと**:
- ❌ ビジネスロジックを実装
- ❌ トランザクション制御（Serviceの仕事）

**実装（`MemoMapper.java` のシグネチャ）**:
```java
@Mapper
public interface MemoMapper {
    List<Memo> findAll();
    Optional<Memo> findById(Long id);       // 「無いかもしれない」を型で表現
    void insert(Memo memo);
    int update(Memo memo);                  // 戻り値 = 更新件数
    int deleteById(Long id);
    List<Memo> searchByKeyword(@Param("keyword") String keyword);
    // 以下はページング実装演習用（現在未使用）
    List<Memo> findAllWithPaging(@Param("limit") int limit, @Param("offset") int offset);
    long count();
}
```

### 4.2 DTOとEntityの使い分け

#### 4.2.1 Entity（エンティティ）

**用途**: データベーステーブルの1行を表現

**特徴**:
- データベースのカラムと1対1で対応
- バリデーションアノテーションは**付けない**
- MyBatisで取得したデータを格納

#### 4.2.2 DTO（データ転送オブジェクト）

**用途**: 画面とサービス層の間でデータ転送

**特徴**:
- バリデーションアノテーションを付ける
- 画面に必要なフィールドのみ含む（Entityと完全に一致しなくてもOK）
- このアプリでは日時を**フォーマット済み文字列**（`createdAtFormatted`）で持ち、テンプレート側のフォーマット処理を不要にしている

#### 4.2.3 変換方法（実装より）

変換ロジックは `MemoDto` 自身が持ちます（Serviceに変換コードが散らばらない）:

```java
// Entity → DTO（日時フォーマット込み）
MemoDto dto = MemoDto.fromEntity(memo);

// DTO → Entity（id・title・contentのみコピー。日時はDBが管理するためコピーしない）
Memo memo = dto.toEntity();
```

## 5. コーディング規約

### 5.1 命名規則

#### 5.1.1 クラス名

| 種類 | 命名規則 | このアプリでの例 |
|------|---------|---|
| Controller | `名詞 + Controller` | `MemoController` |
| Service | `名詞 + Service` | `MemoService`（具象クラス。Impl分離はしていない） |
| Mapper | `名詞 + Mapper` | `MemoMapper` |
| Entity | `名詞` | `Memo` |
| DTO | `名詞 + Dto` | `MemoDto` |
| Exception | `説明 + Exception` | `MemoNotFoundException` |

※ 実務では `MemoService`（インターフェース）+ `MemoServiceImpl`（実装）と分離する流儀もあります。複数実装やAOPが必要になったときに導入を検討してください。

#### 5.1.2 メソッド名

| 操作 | 命名規則 | このアプリでの例 |
|------|---------|---|
| 取得（複数） | `findAll`, `findBy~` | `findAll()` |
| 取得（単一） | `findById` | `findById(Long id)` |
| 作成 | `create`, `insert` | `create(MemoDto dto)` |
| 更新 | `update` | `update(MemoDto dto)` |
| 削除 | `delete`, `deleteById` | `delete(Long id)` / `deleteById(Long id)` |
| 検索 | `search`, `searchBy~` | `search(String keyword)` / `searchByKeyword(...)` |
| カウント | `count` | `count()` |

#### 5.1.3 変数名

```java
// 良い例
List<Memo> memos = memoMapper.findAll();
MemoDto memoDto = new MemoDto();

// 悪い例
List<Memo> m = memoMapper.findAll();        // 短すぎる
MemoDto data = new MemoDto();               // 抽象的すぎる
```

### 5.2 コメントの書き方

このリポジトリは学習用のため、**教育目的の詳細な日本語コメントを書くのが仕様**です。ただし量ではなく質で価値を出します（詳細はリポジトリの学習コメント規約参照）。

**書く価値があるもの**: Why（選定理由）/ 仕組み（フレームワークが裏でやること）/ 落とし穴 / モダン版との対比

```java
// 良い例: 「なぜ」を説明
// トリガーで自動更新されるため、updated_atはSQLに含めない
UPDATE memos SET title = #{title}, content = #{content} WHERE id = #{id}

// 悪い例: コードを読めばわかること（言い換えコメント）
// titleを設定する
memo.setTitle(memoDto.getTitle());
```

**実装と食い違ったコメントはバグと同格**です。コードを変えたらコメントも直してください。

## 6. デバッグ方法

### 6.1 ログ出力

#### 6.1.1 ログレベル（実装での使い分け）

```java
log.debug("メモを取得します: id={}", id);       // 開発時のデバッグ情報
log.info("メモを登録しました: id={}", id);       // ビジネス上の重要イベント
log.warn("存在しないURLへのアクセス: /{}", path); // 正常だが注意すべき状況（404系）
log.error("予期しないエラーが発生しました", ex);  // システムの異常（スタックトレース付き）
```

**このアプリの方針**: 「メモが見つからない」「URLタイプミス」は**システム異常ではない**のでWARN。ERRORは本当の異常だけに使い、監視ノイズを防ぎます（`GlobalExceptionHandler` 参照）。

#### 6.1.2 application.ymlでログレベル設定

```yaml
logging:
  level:
    root: INFO
    com.example.memoapp: DEBUG              # 自分のアプリはDEBUG
    com.example.memoapp.mapper: DEBUG       # SQL実行ログ
```

### 6.3 よく使うデバッグテクニック

#### 6.3.1 SQLの確認

`application.yml` の設定（設定済み）により、実行されるSQLがコンソールに出力されます:

```
==>  Preparing: SELECT id, title, content, created_at, updated_at FROM memos ORDER BY created_at DESC
==> Parameters:
<==      Total: 5
```

## 7. テストの書き方

このリポジトリには**30個の実装済みテスト**があります（`src/test/`）。新しいテストを書くときは、これらを手本にしてください。

### 7.1 テストの種類と実装状況

| テスト種類 | 対象 | ツール | 状態 |
|-----------|------|--------|------|
| 純粋ロジックの単体テスト | DTO変換 | JUnit 5 | 実装済み（`MemoDtoTest`） |
| モックを使う単体テスト | Service | JUnit 5 + Mockito | 実装済み（`MemoServiceTest`） |
| スライステスト | Controller + 例外ハンドラ | @WebMvcTest + MockMvc | 実装済み（`MemoControllerTest`） |
| Mapperテスト | SQL（実DB） | @MybatisTest + Testcontainers | **未実装（学習課題）** |
| E2Eテスト | ブラウザ操作 | Playwright / Selenium | 未実装（高度） |

### 7.2 Serviceテスト（`MemoServiceTest` より抜粋）

```java
@ExtendWith(MockitoExtension.class)     // JUnit 5 に Mockito を組み込む
class MemoServiceTest {

    @Mock                               // MemoMapper のモック（偽物）を生成
    private MemoMapper memoMapper;

    @InjectMocks                        // モックを注入したテスト対象を生成
    private MemoService memoService;

    @Test
    @DisplayName("存在しないIDなら、MemoNotFoundException がスローされる")
    void throwsException_whenMemoDoesNotExist() {
        // Arrange: 「DBに該当行がなかった」状況を再現
        when(memoMapper.findById(999L)).thenReturn(Optional.empty());

        // Act & Assert: 例外がスローされることを検証
        assertThatThrownBy(() -> memoService.findById(999L))
                .isInstanceOf(MemoNotFoundException.class)
                .hasMessageContaining("id=999");
    }
}
```

**ポイント**:
- `@SpringBootTest` は使いません。Service単体のロジック検証にSpringコンテナは不要で、Mockitoだけの方が圧倒的に速い
- 異常系（例外）も仕様の一部としてテストする

### 7.3 Controllerテスト（`MemoControllerTest` より抜粋）

```java
@WebMvcTest(MemoController.class)   // Web層だけを起動するスライステスト
class MemoControllerTest {

    @Autowired
    private MockMvc mockMvc;        // 擬似HTTPリクエストを送る道具

    @MockitoBean                    // DIコンテナ内のServiceをモックに差し替え
    private MemoService memoService;

    @Test
    @DisplayName("存在しないIDなら 404 + エラー画面が返る")
    void returns404_whenMemoNotFound() throws Exception {
        when(memoService.findById(999L))
                .thenThrow(new MemoNotFoundException("メモが見つかりません: id=999"));

        mockMvc.perform(get("/memos/999"))
                .andExpect(status().isNotFound())
                .andExpect(view().name("error/error"));
    }
}
```

**ポイント（落とし穴）**: Spring Boot 3.4以降、`@MockBean` は非推奨になり **`@MockitoBean`**（`org.springframework.test.context.bean.override.mockito` パッケージ）に置き換えられました。古い記事のコードをコピーすると非推奨警告が出ます。

### 7.4 テストの実行

```bash
# 全てのテストを実行
mvn test

# 特定のテストクラスだけ実行
mvn test -Dtest=MemoServiceTest

# 特定のテストメソッドだけ実行（@Nestedクラス内はクラス名も指定）
mvn test -Dtest='MemoServiceTest$FindById'
```

### 7.5 テストのベストプラクティス

**テスト名の付け方**:
- `@DisplayName` で日本語の説明を付ける（テスト結果が仕様書のように読める）
- `@Nested` でメソッドごとにグループ化する

**テストの構造（AAA パターン）**:
1. **Arrange（準備）**: テストに必要なデータやモックの動作を準備
2. **Act（実行）**: テスト対象のメソッドを実行
3. **Assert（検証）**: 期待する結果と実際の結果を比較

**回帰テストの考え方**:
バグを直したら、**その再発を防ぐテスト**を残す。このリポジトリでは「存在しないIDで200が返っていたバグ」「編集エラー時に日時欄が空になるバグ」の回帰テストが `MemoControllerTest` にあります。

**カバレッジの目安**:
- Serviceクラス: 80%以上（ビジネスロジックの主要部分）
- Controller: 主要な画面遷移とバリデーション、エラーパス
- Mapper: 複雑なSQLのみ（基本的なCRUDは省略可）

### 7.6 Mapperテスト（学習課題）

Mapperテストは実際のDBが必要です。本プロジェクトのDDLはPostgreSQL専用構文（BIGSERIAL・トリガー等）のため、**H2での代替はできません**。

実務で一般的な構成は `@MybatisTest` + **Testcontainers**（テスト実行時に使い捨てのPostgreSQLコンテナを自動起動するライブラリ）です。挑戦する場合は「Testcontainers PostgreSQL JUnit5」で調べてみてください。

## 8. よくある問題と解決方法

### 8.1 コンパイルエラー

#### 問題: `Cannot resolve symbol` (Lombok関連)

**原因**: Lombok プラグインがインストールされていない

**解決方法**:
```
1. IntelliJ IDEA → Settings → Plugins
2. "Lombok" を検索してインストール
3. 再起動
```

#### 問題: `Cannot resolve method 'xxx'`

**原因**: Mavenの依存関係がダウンロードされていない

**解決方法**:
```
IntelliJ IDEA の右サイドバー → Maven → Reload All Maven Projects
```

### 8.2 実行時エラー

#### 問題: `Port 8080 is already in use`

**解決方法**:
```yaml
# application.yml
server:
  port: 8081  # 別のポートに変更
```

#### 問題: `Connection refused` (PostgreSQL)

**解決方法**:
```bash
docker compose up -d postgres
docker compose ps  # 起動確認
```

#### 問題: `Flyway migration failed` / `Migration checksum mismatch`

**原因**: 適用済みマイグレーションファイルの変更（コメント変更でもchecksumが変わる）、またはSQL文法エラー

**解決方法**:
```bash
# 学習環境ではDBリセットが最も確実（データは消えます）
docker compose down -v
docker compose up -d postgres
mvn spring-boot:run
```
※ `mvn flyway:repair` 等のMavenコマンドは flyway-maven-plugin を導入していない本プロジェクトでは**使えません**。マイグレーションはアプリ起動時に自動実行されます。

### 8.3 画面表示エラー

#### 問題: Thymeleafで `null` が表示される / エラーになる

**原因**: Controllerでモデルに値を設定していない

**解決方法**:
```java
@GetMapping("")
public String list(Model model) {
    model.addAttribute("memos", memoService.search(null));  // ← これを忘れずに！
    return "memos/list";
}
```

#### 問題: CSSが反映されない

**解決方法**:
```
ブラウザで Ctrl+F5 (強制リロード)
```

#### 問題: エラー画面は出るのにHTTPステータスが200になる

**原因**: `@ExceptionHandler` メソッドに `@ResponseStatus` を付けていない（例外クラス側の指定はハンドラ捕捉時には効かない）

**解決方法**: `GlobalExceptionHandler` の各ハンドラのように、メソッド側に `@ResponseStatus` を明示する

## 9. Git運用

このリポジトリでは **main へ直接コミットせず、ブランチ + Pull Request** で作業します（1 PR = 1 関心事。レビューの練習も学習の一部）。

### 9.1 基本的な作業フロー

```bash
# 1. mainを最新化
git checkout main
git pull

# 2. 作業ブランチを作成（feature/ fix/ docs/ のプレフィックスを付ける）
git checkout -b feature/add-paging

# 3. 開発してコミット
git add .
git commit -m "feat: 一覧画面にページングを追加"

# 4. リモートにプッシュしてPull Requestを作成
git push -u origin feature/add-paging
# → GitHubの画面（または gh pr create）でPRを作成し、セルフレビューしてからマージ
```

### 9.2 コミットメッセージの書き方

#### 9.2.1 プレフィックス

| プレフィックス | 意味 | 例 |
|--------------|------|---|
| `feat:` | 新機能 | `feat: メモ検索機能を追加` |
| `fix:` | バグ修正 | `fix: 削除時のNullPointerExceptionを修正` |
| `docs:` | ドキュメント | `docs: READMEに環境構築手順を追加` |
| `refactor:` | リファクタリング | `refactor: Service層のコードを整理` |
| `test:` | テスト | `test: MemoServiceのテストを追加` |
| `chore:` | その他 | `chore: pom.xmlの依存関係を更新` |

#### 9.2.2 良いコミットメッセージ

```
✅ 良い例
feat: メモにカテゴリー機能を追加
fix: タイトル未入力時のバリデーションエラーを修正

❌ 悪い例
update
修正
バグ直した
```

### 9.3 ブランチの命名

| プレフィックス | 用途 | 例 |
|--------------|------|---|
| `feature/` | 機能追加 | `feature/add-paging` |
| `fix/` | バグ修正 | `fix/delete-dialog-duplication` |
| `docs/` | ドキュメント | `docs/update-setup-guide` |
