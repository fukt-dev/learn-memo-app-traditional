# メモ帳アプリケーション 設計書（従来型）

## 1. システム概要

### 1.1 目的
本アプリケーションは、開発未経験者がWebアプリケーション開発の基礎を学ぶための教材として、基本的なCRUD操作を実装したメモ帳アプリケーションです。従来型のサーバーサイドレンダリング（MVC）構成を採用し、3層レイヤーアーキテクチャで実装します。

**実務を意識した技術選定**:
- **PostgreSQL**: エンタープライズ開発で広く採用されているRDBMS
- **MyBatis + 2waySQL**: 日本の業務システム開発で標準的に使われる技術
- **Flyway**: データベースマイグレーション管理（実務必須スキル）

### 1.2 対象ユーザー
- Webアプリケーション開発の学習を始めたばかりの初心者
- 従来型のMVCアーキテクチャを学びたい開発者
- Java/Spring Bootの基礎を学びたい開発者
- 実務で使われる2waySQL（MyBatis）を習得したい開発者

### 1.3 主な機能
- メモの新規作成
- メモの一覧表示
- メモの詳細表示
- メモの編集
- メモの削除
- メモの検索（オプション）

## 2. アーキテクチャ設計

### 2.1 3層レイヤーアーキテクチャ

本アプリケーションは、以下の3層構造で実装します：

```
┌─────────────────────────────────────┐
│    プレゼンテーション層              │
│  (Presentation Layer)               │
│  - Controller                       │
│  - Thymeleaf (View)                 │
│  - JavaScript/jQuery (動的な操作)   │
└─────────────────────────────────────┘
              ↓↑
┌─────────────────────────────────────┐
│    ビジネスロジック層                │
│  (Business Logic Layer)             │
│  - Service                          │
│  - DTO (Data Transfer Object)       │
└─────────────────────────────────────┘
              ↓↑
┌─────────────────────────────────────┐
│    データアクセス層                  │
│  (Data Access Layer)                │
│  - Mapper (MyBatis)                 │
│  - SQL (2waySQL in XML)             │
│  - Entity                           │
│  - Database (PostgreSQL)            │
└─────────────────────────────────────┘
```

#### 2.1.1 プレゼンテーション層
- **責務**: ユーザーとのインタラクション、画面表示、入力値の受け取り
- **主要コンポーネント**:
  - **Controller**: HTTPリクエストを受け取り、Serviceを呼び出し、Viewに渡すデータを準備
  - **Thymeleaf**: HTMLを生成するテンプレートエンジン
  - **JavaScript/jQuery**: クライアントサイドでの動的な操作（確認ダイアログ、バリデーション表示など）

#### 2.1.2 ビジネスロジック層
- **責務**: ビジネスルールの実装、トランザクション管理、データの加工
- **主要コンポーネント**:
  - **Service**: ビジネスロジックを実装するクラス
  - **DTO**: 層間でデータを受け渡すためのオブジェクト

#### 2.1.3 データアクセス層
- **責務**: データベースへのアクセス、永続化処理
- **主要コンポーネント**:
  - **Mapper**: MyBatisのMapperインターフェース（データベース操作の定義）
  - **SQL**: XMLファイルに記述された2waySQL
  - **Entity**: データベーステーブルに対応するJavaオブジェクト
  - **Database**: PostgreSQL（実際のデータ格納先）

### 2.2 技術選定の理由

#### なぜPostgreSQLか？
1. **エンタープライズでの採用実績**: 金融・公共システムなどで広く使われている
2. **機能の豊富さ**: トランザクション、JSON型、全文検索など
3. **標準SQL準拠**: SQL標準に忠実で移植性が高い
4. **オープンソース**: 商用利用も可能でライセンスコストがかからない

#### なぜMyBatis + 2waySQLか？
1. **実務での標準**: 日本の業務システム開発で広く採用されている
2. **SQLの可視性**: SQLを直接記述するため、パフォーマンスチューニングが容易
3. **複雑なクエリへの対応**: ORMでは表現しにくい複雑なJOINやサブクエリも自由に記述
4. **2waySQL**: 動的なSQLとテスト可能な静的SQLを両立
5. **TERASOLUNA準拠**: TERASOLUNAフレームワークでも標準採用

#### なぜFlywayによるマイグレーション管理か？
1. **SQLの学習機会**: マイグレーションファイルを手書きするため、SQL学習が阻害されない
2. **バージョン管理**: データベーススキーマの変更履歴を管理できる
3. **チーム開発**: 開発者間でデータベース構造を同期できる
4. **環境構築の簡素化**: 初期セットアップが容易になり、アプリケーション開発に集中できる
5. **実務スキル**: 実際のプロジェクトで必須の技術

**重要**: `ddl-auto: update` のような完全自動生成は使用せず、**マイグレーションファイルを手書き**します。

#### なぜサーバーサイドレンダリング（MVC）か？
1. **シンプルさ**: クライアント側の複雑な状態管理が不要
2. **学習曲線**: フロントエンドフレームワークの習得が不要
3. **従来型の理解**: モダン構成との比較学習のベースライン

## 3. 機能要件

### 3.1 機能一覧

| 機能ID | 機能名 | 優先度 | 説明 |
|--------|--------|--------|------|
| F001 | メモ一覧表示 | 必須 | 登録されているすべてのメモを一覧表示する |
| F002 | メモ詳細表示 | 必須 | 選択したメモの詳細を表示する |
| F003 | メモ新規作成 | 必須 | 新しいメモを作成する |
| F004 | メモ編集 | 必須 | 既存のメモを編集する |
| F005 | メモ削除 | 必須 | 既存のメモを削除する（削除確認あり） |
| F006 | メモ検索 | 推奨 | タイトルまたは内容でメモを検索する |
| F007 | ページング | オプション | 一覧表示時にページング機能を提供する |
| F008 | カテゴリー分類 | オプション | メモをカテゴリーで分類する |

### 3.2 機能詳細仕様

#### F001: メモ一覧表示
- **入力**: なし
- **処理**:
  1. データベースからすべてのメモを取得
  2. 作成日時の降順（新しい順）でソート
  3. 一覧画面に表示
- **出力**: メモ一覧（ID、タイトル、作成日時、更新日時）
- **例外処理**: データベース接続エラー時はエラーメッセージを表示

#### F002: メモ詳細表示
- **入力**: メモID
- **処理**:
  1. 指定されたIDのメモをデータベースから取得
  2. 詳細画面に表示
- **出力**: メモ詳細（ID、タイトル、内容、作成日時、更新日時）
- **例外処理**:
  - メモが存在しない場合: 404エラー
  - 不正なID形式: バリデーションエラー

#### F003: メモ新規作成
- **入力**: タイトル、内容
- **処理**:
  1. 入力値のバリデーション
  2. 現在日時を作成日時・更新日時として設定
  3. データベースに保存
  4. 一覧画面にリダイレクト
- **出力**: 作成成功メッセージ、一覧画面への遷移
- **バリデーション**:
  - タイトル: 必須、最大100文字
  - 内容: 任意、最大5000文字
- **例外処理**: バリデーションエラー時は入力値を保持して再表示

#### F004: メモ編集
- **入力**: メモID、タイトル、内容
- **処理**:
  1. 指定されたIDのメモをデータベースから取得
  2. 入力値のバリデーション
  3. 更新日時を現在日時に更新
  4. データベースを更新
  5. 詳細画面にリダイレクト
- **出力**: 更新成功メッセージ、詳細画面への遷移
- **バリデーション**: F003と同じ
- **例外処理**:
  - メモが存在しない場合: 404エラー
  - バリデーションエラー時は入力値を保持して再表示

#### F005: メモ削除
- **入力**: メモID
- **処理**:
  1. JavaScriptで削除確認ダイアログを表示
  2. ユーザーがOKを押した場合のみ削除リクエストを送信
  3. 指定されたIDのメモをデータベースから削除
  4. 一覧画面にリダイレクト
- **出力**: 削除成功メッセージ、一覧画面への遷移
- **例外処理**: メモが存在しない場合でもエラーとせず、一覧画面にリダイレクト

#### F006: メモ検索（推奨機能）
- **入力**: 検索キーワード
- **処理**:
  1. タイトルまたは内容に検索キーワードを含むメモを取得（2waySQL使用）
  2. 一覧画面に検索結果を表示
- **出力**: 検索結果のメモ一覧
- **例外処理**: 検索結果が0件の場合は「該当するメモがありません」を表示

## 4. 非機能要件

### 4.1 パフォーマンス要件
- 一覧画面の初期表示: 3秒以内
- CRUD操作のレスポンス: 1秒以内
- 同時アクセス: 10ユーザーまで（学習用のため）

### 4.2 可用性要件
- 学習用のため、高可用性は求めない
- ローカル環境での動作を前提

### 4.3 セキュリティ要件
- XSS対策: テンプレートエンジンのエスケープ機能を使用
- SQLインジェクション対策: MyBatisのプリペアドステートメント使用
- CSRF対策: Spring Securityのデフォルト設定を使用（将来的に）
- 認証・認可: 現バージョンでは実装しない（将来の拡張として）

### 4.4 保守性要件
- コードの可読性: 初心者が理解できるシンプルなコード
- コメント: 重要な処理には日本語コメントを記載
- ログ出力: エラー発生時の原因特定が容易なログ設計
- SQLの可視性: 2waySQLによりSQLが明示的で保守しやすい

### 4.5 移植性要件
- OS: Windows、macOS、Linuxで動作
- データベース: PostgreSQL 12以上
- Java: Java 17以上
- ブラウザ: Chrome、Firefox、Edge、Safari（最新版）

### 5.4 DDL（Flywayマイグレーションファイル）

#### V1__create_memos_table.sql
```sql
-- memosテーブル作成
-- PostgreSQL用DDL

CREATE TABLE memos (
    id BIGSERIAL PRIMARY KEY,
    title VARCHAR(100) NOT NULL,
    content TEXT,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 作成日時のインデックス（一覧表示のソート用）
CREATE INDEX idx_memos_created_at ON memos(created_at DESC);

-- 更新日時を自動更新するトリガー関数
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 更新日時自動更新のトリガー
CREATE TRIGGER update_memos_updated_at
    BEFORE UPDATE ON memos
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at_column();

-- テーブルコメント
COMMENT ON TABLE memos IS 'メモ情報を管理するテーブル';
COMMENT ON COLUMN memos.id IS 'メモID（主キー）';
COMMENT ON COLUMN memos.title IS 'メモのタイトル';
COMMENT ON COLUMN memos.content IS 'メモの内容';
COMMENT ON COLUMN memos.created_at IS '作成日時';
COMMENT ON COLUMN memos.updated_at IS '更新日時';
```

```sql
-- 全文検索用のインデックス（PostgreSQLのGINインデックス）
-- オプション機能：検索機能を実装する場合に使用

-- 日本語全文検索用の拡張機能を有効化
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- トライグラムインデックスを作成（部分一致検索の高速化）
CREATE INDEX idx_memos_title_trgm ON memos USING gin (title gin_trgm_ops);

```

### 5.5 PostgreSQL特有の機能

#### トリガーによる更新日時の自動更新
- MySQLの `ON UPDATE CURRENT_TIMESTAMP` に相当する機能をトリガーで実装
- `update_updated_at_column()` 関数を作成し、UPDATE時に自動実行

#### BIGSERIAL型
- PostgreSQLの自動採番型
- MySQLの `BIGINT AUTO_INCREMENT` に相当

#### トライグラムインデックス（pg_trgm拡張）
- 日本語を含む部分一致検索の高速化
- LIKE検索のパフォーマンス向上

### 7.2 パッケージ構成の説明

#### controller（プレゼンテーション層）
- HTTPリクエストを受け取り、Serviceを呼び出す
- ビジネスロジックは持たない
- Viewに渡すデータの準備のみを行う

#### service（ビジネスロジック層）
- アプリケーションのビジネスルールを実装
- トランザクション境界を管理（@Transactional）
- Mapperを呼び出してデータを永続化

#### mapper（データアクセス層）
- **MyBatisのMapperインターフェース**
- データベースへのアクセスを抽象化
- 実際のSQLはXMLファイルに記述（2waySQL）

#### entity（エンティティ）
- データベーステーブルに対応するJavaクラス
- MyBatisで取得したデータをマッピング

#### dto（データ転送オブジェクト）
- 層間でデータを受け渡すためのクラス
- Entityとは別に定義することで、層間の独立性を保つ

#### mybatis/mapper（SQLマッパーXML）
- **2waySQLを記述するXMLファイル**
- Mapperインターフェースと対応
- 動的SQLと静的SQLを両立

#### MemoMapper.xml（2waySQL）
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">

<!-- メモMapperのSQLマッピング -->
<mapper namespace="com.example.memoapp.mapper.MemoMapper">

    <!-- 結果マッピング定義 -->
    <resultMap id="memoResultMap" type="com.example.memoapp.entity.Memo">
        <id property="id" column="id"/>
        <result property="title" column="title"/>
        <result property="content" column="content"/>
        <result property="createdAt" column="created_at"/>
        <result property="updatedAt" column="updated_at"/>
    </resultMap>

    <!-- すべてのメモを作成日時降順で取得 -->
    <select id="findAllOrderByCreatedAtDesc" resultMap="memoResultMap">
        SELECT
            id,
            title,
            content,
            created_at,
            updated_at
        FROM
            memos
        ORDER BY
            created_at DESC
    </select>

    <!-- IDでメモを取得 -->
    <select id="findById" resultMap="memoResultMap">
        SELECT
            id,
            title,
            content,
            created_at,
            updated_at
        FROM
            memos
        WHERE
            id = #{id}
    </select>

    <!--
        キーワード検索（2waySQL）
        キーワードがnullまたは空文字の場合は全件取得
        動的SQLとして機能しつつ、単体でもテスト可能
    -->
    <select id="searchByKeyword" resultMap="memoResultMap">
        SELECT
            id,
            title,
            content,
            created_at,
            updated_at
        FROM
            memos
        <where>
            <!-- キーワードが指定されている場合のみ検索条件を追加 -->
            <if test="keyword != null and keyword != ''">
                AND (
                    title LIKE CONCAT('%', #{keyword}, '%')
                    OR content LIKE CONCAT('%', #{keyword}, '%')
                )
            </if>
        </where>
        ORDER BY
            created_at DESC
    </select>

    <!-- メモを新規作成 -->
    <insert id="insert" useGeneratedKeys="true" keyProperty="id">
        INSERT INTO memos (
            title,
            content,
            created_at,
            updated_at
        ) VALUES (
            #{title},
            #{content},
            CURRENT_TIMESTAMP,
            CURRENT_TIMESTAMP
        )
    </insert>

    <!-- メモを更新 -->
    <update id="update">
        UPDATE memos
        SET
            title = #{title},
            content = #{content}
            -- updated_atはトリガーで自動更新されるため不要
        WHERE
            id = #{id}
    </update>

    <!-- メモを削除 -->
    <delete id="deleteById">
        DELETE FROM memos
        WHERE id = #{id}
    </delete>

</mapper>
```

**2waySQLの特徴**:
1. **可読性**: SQLが明示的で理解しやすい
2. **テスタビリティ**: SQLを単体でテスト可能
3. **動的SQL**: `<if>`タグで条件分岐
4. **パフォーマンス**: 複雑なクエリも最適化しやすい

#### MemoServiceImpl.java
```java
package com.example.memoapp.service.impl;

import com.example.memoapp.dto.MemoDto;
import com.example.memoapp.entity.Memo;
import com.example.memoapp.exception.MemoNotFoundException;
import com.example.memoapp.mapper.MemoMapper;
import com.example.memoapp.service.MemoService;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import java.util.List;
import java.util.stream.Collectors;

/**
 * メモサービス実装クラス
 */
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class MemoServiceImpl implements MemoService {

    private final MemoMapper memoMapper;

    @Override
    public List<MemoDto> findAll() {
        return memoMapper.findAllOrderByCreatedAtDesc()
                .stream()
                .map(this::convertToDto)
                .collect(Collectors.toList());
    }

    @Override
    public MemoDto findById(Long id) {
        Memo memo = memoMapper.findById(id);
        if (memo == null) {
            throw new MemoNotFoundException("メモが見つかりません。ID: " + id);
        }
        return convertToDto(memo);
    }

    @Override
    public List<MemoDto> searchByKeyword(String keyword) {
        return memoMapper.searchByKeyword(keyword)
                .stream()
                .map(this::convertToDto)
                .collect(Collectors.toList());
    }

    @Override
    @Transactional
    public MemoDto create(MemoDto memoDto) {
        Memo memo = convertToEntity(memoDto);
        memoMapper.insert(memo);
        // insertでIDが自動設定される（useGeneratedKeys）
        return convertToDto(memo);
    }

    @Override
    @Transactional
    public MemoDto update(Long id, MemoDto memoDto) {
        Memo existingMemo = memoMapper.findById(id);
        if (existingMemo == null) {
            throw new MemoNotFoundException("メモが見つかりません。ID: " + id);
        }

        existingMemo.setTitle(memoDto.getTitle());
        existingMemo.setContent(memoDto.getContent());

        memoMapper.update(existingMemo);

        // 更新後のデータを再取得して返す
        Memo updatedMemo = memoMapper.findById(id);
        return convertToDto(updatedMemo);
    }

    @Override
    @Transactional
    public void delete(Long id) {
        memoMapper.deleteById(id);
    }

    /**
     * EntityをDTOに変換
     */
    private MemoDto convertToDto(Memo memo) {
        return new MemoDto(
                memo.getId(),
                memo.getTitle(),
                memo.getContent(),
                memo.getCreatedAt(),
                memo.getUpdatedAt()
        );
    }

    /**
     * DTOをEntityに変換
     */
    private Memo convertToEntity(MemoDto dto) {
        Memo memo = new Memo();
        memo.setTitle(dto.getTitle());
        memo.setContent(dto.getContent());
        return memo;
    }
}
```

## 14. MyBatis 2waySQL解説

### 14.1 2waySQLとは

**2waySQL**とは、MyBatisで使われるSQL記述方式で、以下の特徴があります：

1. **動的SQLとして機能**: アプリケーション実行時に条件分岐が可能
2. **静的SQLとしてテスト可能**: SQLツールで直接実行してテスト可能
3. **可読性が高い**: SQLが明示的で保守しやすい

### 14.2 2waySQLの例

```xml
<!-- 検索条件に応じて動的にWHERE句を構築 -->
<select id="searchByKeyword" resultMap="memoResultMap">
    SELECT
        id,
        title,
        content,
        created_at,
        updated_at
    FROM
        memos
    <where>
        <if test="keyword != null and keyword != ''">
            AND (
                title LIKE CONCAT('%', #{keyword}, '%')
                OR content LIKE CONCAT('%', #{keyword}, '%')
            )
        </if>
    </where>
    ORDER BY
        created_at DESC
</select>
```

**動作**:
- `keyword`が指定されている場合: `WHERE`句が追加され、LIKE検索が実行される
- `keyword`がnullまたは空: `WHERE`句は追加されず、全件取得

**テスト方法**:
```sql
-- SQLツールで直接実行可能（動的部分を手動で削除）
SELECT
    id,
    title,
    content,
    created_at,
    updated_at
FROM
    memos
WHERE
    title LIKE '%学習%' OR content LIKE '%学習%'
ORDER BY
    created_at DESC;
```

### 14.3 2waySQLのメリット

1. **パフォーマンスチューニングが容易**
   - SQLが明示的なので、実行計画を確認しやすい
   - インデックスの効果を検証しやすい

2. **複雑なクエリに対応**
   - JOINやサブクエリも自由に記述可能
   - ORMでは表現しにくいSQLも対応可能

3. **保守性が高い**
   - SQLの変更がXMLファイルで完結
   - Javaコードとの分離で見通しが良い

4. **学習しやすい**
   - SQLの基礎知識があれば理解できる
   - 動的SQLの仕組みが明確

### 16.2 つまづきやすいポイント

#### 環境構築
- **問題**: JDKのバージョンが合わない
- **対処**: Java 17以上をインストール、JAVA_HOMEを確認

#### PostgreSQL接続
- **問題**: 接続エラーが発生する
- **対処**:
  - application.ymlの設定を確認
  - PostgreSQLが起動しているか確認
  - ユーザー名・パスワードが正しいか確認

#### Flywayマイグレーション
- **問題**: マイグレーションが失敗する
- **対処**:
  - SQLファイルの文法エラーを確認
  - データベースが空であることを確認
  - Flywayのログを確認

#### MyBatisのマッピング
- **問題**: SQLの実行結果がnullになる
- **対処**:
  - resultMapの定義を確認
  - カラム名とプロパティ名のマッピングを確認
  - map-underscore-to-camel-caseの設定を確認

#### Thymeleafの文法
- **問題**: テンプレートエンジンの書き方がわからない
- **対処**: 公式ドキュメントを参照、既存のテンプレートを参考にする

#### バリデーションエラー
- **問題**: エラーメッセージが表示されない
- **対処**: BindingResultの配置順序、Thymeleafでのエラー表示を確認

## 付録

### A. 用語集

- **CRUD**: Create（作成）、Read（読み取り）、Update（更新）、Delete（削除）の頭文字
- **MVC**: Model-View-Controller の略。MVCパターンはアプリケーションを3つの役割に分離する設計パターン
- **DTO**: Data Transfer Object。層間でデータを受け渡すためのオブジェクト
- **Entity**: データベーステーブルに対応するJavaオブジェクト
- **Mapper**: MyBatisのデータアクセスインターフェース（RepositoryパターンのMyBatis版）
- **2waySQL**: 動的SQLとしても静的SQLとしても利用できるSQL記述方式
- **DI**: Dependency Injection（依存性注入）。オブジェクトの依存関係を外部から注入する設計パターン
- **O/Rマッパー**: Object-Relational Mapper。オブジェクトとリレーショナルデータベースをマッピングする技術

### B. トラブルシューティング

```bash

netstat -ano | findstr :8080
taskkill /PID <PID> /F
```
