## 1. システム概要

### 1.1 目的

本アプリケーションは、開発未経験者がWebアプリケーション開発の基礎を学ぶための教材として、基本的なCRUD操作を実装したメモ帳アプリケーションです。従来型のサーバーサイドレンダリング（MVC）構成を採用し、3層レイヤーアーキテクチャで実装しています。

**実務を意識した技術選定**:
- **PostgreSQL**: エンタープライズ開発で広く採用されているRDBMS
- **MyBatis + 2waySQL**: 日本の業務システム開発で標準的に使われる技術
- **Flyway**: データベースマイグレーション管理（実務必須スキル）

### 1.2 対象ユーザー

- Webアプリケーション開発の学習を始めたばかりの初心者
- 従来型のMVCアーキテクチャを学びたい開発者
- Java/Spring Bootの基礎を学びたい開発者
- 実務で使われる2waySQL（MyBatis）を習得したい開発者

### 1.3 主な機能

| 機能 | 状態 |
|------|------|
| メモの新規作成 | 実装済み |
| メモの一覧表示 | 実装済み |
| メモの詳細表示 | 実装済み |
| メモの編集 | 実装済み |
| メモの削除（確認ダイアログあり） | 実装済み |
| メモの検索（キーワード部分一致） | 実装済み |
| ページング | 未実装（Service/Mapperの部品のみ用意。実装演習） |

## 2. アーキテクチャ設計

### 2.1 3層レイヤーアーキテクチャ

```
┌─────────────────────────────────────┐
│    プレゼンテーション層              │
│  (Presentation Layer)               │
│  - Controller                       │
│  - Thymeleaf (View)                 │
│  - JavaScript (Vanilla)             │
└─────────────────────────────────────┘
              ↓↑
┌─────────────────────────────────────┐
│    ビジネスロジック層                │
│  (Business Logic Layer)             │
│  - Service                          │
│  - DTO (Data Transfer Object)       │
└─────────────────────────────────────┘
              ↓↑
┌─────────────────────────────────────┐
│    データアクセス層                  │
│  (Data Access Layer)                │
│  - Mapper (MyBatis)                 │
│  - SQL (2waySQL in XML)             │
│  - Entity                           │
│  - Database (PostgreSQL)            │
└─────────────────────────────────────┘
```

#### 2.1.1 プレゼンテーション層
- **責務**: ユーザーとのインタラクション、画面表示、入力値の受け取り
- **主要コンポーネント**:
  - **Controller**: HTTPリクエストを受け取り、Serviceを呼び出し、Viewに渡すデータを準備
  - **Thymeleaf**: HTMLを生成するテンプレートエンジン。共通部品は `th:fragment` / `th:replace` で共通化
  - **JavaScript (Vanilla)**: フレームワークやjQueryを使わない素のJavaScript。削除確認ダイアログ・クライアントサイドバリデーション・フラッシュメッセージの自動非表示を実装

#### 2.1.2 ビジネスロジック層
- **責務**: ビジネスルールの実装、トランザクション管理、データの加工
- **主要コンポーネント**:
  - **Service**: ビジネスロジックを実装するクラス
  - **DTO**: 層間でデータを受け渡すためのオブジェクト

#### 2.1.3 データアクセス層
- **責務**: データベースへのアクセス、永続化処理
- **主要コンポーネント**:
  - **Mapper**: MyBatisのMapperインターフェース（データベース操作の定義）
  - **SQL**: XMLファイルに記述された2waySQL
  - **Entity**: データベーステーブルに対応するJavaオブジェクト
  - **Database**: PostgreSQL（実際のデータ格納先）

### 2.2 技術選定の理由

#### なぜPostgreSQLか？
1. **エンタープライズでの採用実績**: 金融・公共システムなどで広く使われている
2. **機能の豊富さ**: トランザクション、JSON型、全文検索など
3. **標準SQL準拠**: SQL標準に忠実で移植性が高い
4. **オープンソース**: 商用利用も可能でライセンスコストがかからない

#### なぜMyBatis + 2waySQLか？
1. **実務での標準**: 日本の業務システム開発で広く採用されている
2. **SQLの可視性**: SQLを直接記述するため、パフォーマンスチューニングが容易
3. **複雑なクエリへの対応**: ORMでは表現しにくい複雑なJOINやサブクエリも自由に記述
4. **2waySQL**: 動的なSQLとテスト可能な静的SQLを両立

#### なぜFlywayによるマイグレーション管理か？
1. **SQLの学習機会**: マイグレーションファイルを手書きするため、SQL学習が阻害されない
2. **バージョン管理**: データベーススキーマの変更履歴を管理できる
3. **チーム開発**: 開発者間でデータベース構造を同期できる
4. **実務スキル**: 実際のプロジェクトで必須の技術

**重要**: `ddl-auto: update` のような完全自動生成は使用せず、**マイグレーションファイルを手書き**します。
マイグレーションは**アプリケーション起動時に自動実行**されます（Mavenコマンドによる手動実行は本プロジェクトでは構成していません。詳細は[10. 設定ファイル](#10-設定ファイル)を参照）。

#### なぜサーバーサイドレンダリング（MVC）か？
1. **シンプルさ**: クライアント側の複雑な状態管理が不要
2. **学習曲線**: フロントエンドフレームワークの習得が不要
3. **従来型の理解**: モダン構成（[learn-memo-app-modern](https://github.com/fukt-dev/learn-memo-app-modern)）との比較学習のベースライン

## 3. 機能要件

### 3.1 機能一覧

| 機能ID | 機能名 | 状態 | 説明 |
|--------|--------|------|------|
| F001 | メモ一覧表示 | 実装済み | 登録されているすべてのメモを一覧表示する |
| F002 | メモ詳細表示 | 実装済み | 選択したメモの詳細を表示する |
| F003 | メモ新規作成 | 実装済み | 新しいメモを作成する |
| F004 | メモ編集 | 実装済み | 既存のメモを編集する |
| F005 | メモ削除 | 実装済み | 既存のメモを削除する（削除確認あり） |
| F006 | メモ検索 | 実装済み | タイトルまたは内容でメモを検索する |
| F007 | ページング | 実装演習 | 一覧表示時にページング機能を提供する |
| F008 | カテゴリー分類 | 将来拡張 | メモをカテゴリーで分類する |

### 3.2 機能詳細仕様

#### F001: メモ一覧表示
- **入力**: なし
- **処理**:
  1. データベースからすべてのメモを取得
  2. 作成日時の降順（新しい順）でソート
  3. 一覧画面に表示（本文は100文字で省略表示）
- **出力**: メモ一覧（タイトル、作成日時、本文プレビュー）
- **例外処理**: データベース接続エラー時はエラーメッセージを表示（500）

#### F002: メモ詳細表示
- **入力**: メモID（URLパス）
- **処理**:
  1. 指定されたIDのメモをデータベースから取得
  2. 詳細画面に表示
- **出力**: メモ詳細（タイトル、内容、作成日時、更新日時）
- **例外処理**: メモが存在しない場合は404エラー画面を表示

#### F003: メモ新規作成
- **入力**: タイトル、本文
- **処理**:
  1. 入力値のバリデーション
  2. データベースに保存（作成日時・更新日時はDB側のDEFAULTで自動設定）
  3. 一覧画面にリダイレクト（PRGパターン）
- **出力**: 作成成功メッセージ（フラッシュメッセージ）、一覧画面への遷移
- **バリデーション**:
  - タイトル: **必須、1〜200文字**（DBの `VARCHAR(200)` に対応）
  - 本文: **必須**、文字数制限なし（DBの `TEXT` 型に対応）
- **例外処理**: バリデーションエラー時は入力値を保持して再表示

#### F004: メモ編集
- **入力**: メモID（URLパス）、タイトル、本文
- **処理**:
  1. 指定されたIDのメモをデータベースから取得（存在チェック）
  2. 入力値のバリデーション
  3. データベースを更新（更新日時はDBトリガーで自動更新）
  4. 詳細画面にリダイレクト
- **出力**: 更新成功メッセージ、詳細画面への遷移
- **バリデーション**: F003と同じ
- **例外処理**:
  - メモが存在しない場合: 404エラー
  - バリデーションエラー時は入力値を保持して再表示（作成・更新日時はサーバー側で再取得して表示を維持）

#### F005: メモ削除
- **入力**: メモID（URLパス）
- **処理**:
  1. JavaScript（script.js の `initDeleteConfirmation()`）で削除確認ダイアログを表示
  2. ユーザーがOKを押した場合のみ削除リクエストを送信
  3. 指定されたIDのメモをデータベースから削除
  4. 一覧画面にリダイレクト
- **出力**: 削除成功メッセージ、一覧画面への遷移
- **例外処理**: メモが存在しない場合は404エラー

#### F006: メモ検索
- **入力**: 検索キーワード（クエリパラメータ `keyword`）
- **処理**:
  1. タイトルまたは本文にキーワードを含むメモをLIKE部分一致で取得
  2. キーワードが空・未指定の場合は全件取得（判断はService層が担う）
  3. 一覧画面に検索結果を表示
- **出力**: 検索結果のメモ一覧
- **例外処理**: 検索結果が0件の場合は「メモがありません」を表示
- **既知の制限**: LIKEのワイルドカード文字（`%` `_`）はエスケープしていない（詳細は `MemoMapper.xml` のコメント参照）

## 4. 非機能要件

### 4.1 パフォーマンス要件
- 一覧画面の初期表示: 3秒以内
- CRUD操作のレスポンス: 1秒以内
- 同時アクセス: 10ユーザーまで（学習用のため）

### 4.2 可用性要件
- 学習用のため、高可用性は求めない
- ローカル環境での動作を前提

### 4.3 セキュリティ要件
- XSS対策: Thymeleafの自動エスケープ機能を使用
- SQLインジェクション対策: MyBatisのプリペアドステートメント（`#{}` 記法）を使用
- CSRF対策: 未実装（Spring Security導入時の学習課題）
- 認証・認可: 未実装（将来の拡張として）

### 4.4 保守性要件
- コードの可読性: 初心者が理解できるシンプルなコード
- コメント: 教育目的の詳細な日本語コメント（Why・仕組み・落とし穴を中心に記載）
- ログ出力: エラー発生時の原因特定が容易なログ設計（404系はWARN、システム異常はERROR）
- SQLの可視性: 2waySQLによりSQLが明示的で保守しやすい

### 4.5 移植性要件
- OS: Windows、macOS、Linuxで動作
- データベース: PostgreSQL 16（docker-compose.yml は `postgres:16-alpine` を使用）
- Java: Java 17以上
- ブラウザ: Chrome、Firefox、Edge、Safari（最新版）

## 5. データベース設計

### 5.1 ER図

```
┌─────────────────────┐
│      memos          │
├─────────────────────┤
│ id (PK)             │ BIGSERIAL
│ title               │ VARCHAR(200) NOT NULL
│ content             │ TEXT NOT NULL
│ created_at          │ TIMESTAMP NOT NULL
│ updated_at          │ TIMESTAMP NOT NULL
└─────────────────────┘
```

### 5.2 テーブル定義

#### memosテーブル

| カラム名 | データ型 | NULL許可 | デフォルト値 | 説明 |
|----------|----------|----------|--------------|------|
| id | BIGSERIAL | NO | 自動採番 | 主キー、メモID |
| title | VARCHAR(200) | NO | - | メモのタイトル |
| content | TEXT | NO | - | メモの本文 |
| created_at | TIMESTAMP | NO | CURRENT_TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | NO | CURRENT_TIMESTAMP | 更新日時（トリガーで自動更新） |

### 5.3 インデックス設計

| インデックス名 | カラム | 種類 | 目的 |
|---------------|--------|------|------|
| （主キー） | id | PRIMARY KEY | 主キー |
| idx_memos_created_at | created_at DESC | B-tree | 一覧表示時のソート用 |
| idx_memos_fulltext | to_tsvector('simple', title \|\| ' ' \|\| content) | GIN | 全文検索（`@@` 演算子）用 |

**重要（学習ポイント）**: `idx_memos_fulltext` は全文検索演算子（`@@`）専用のインデックスであり、現在の検索機能（LIKE部分一致）からは**使われません**。LIKE検索を高速化するには `pg_trgm` 拡張 + `gin_trgm_ops` のインデックスが必要です（実装演習。`V1__Create_memos_table.sql` のコメント参照）。

### 5.4 DDL（Flywayマイグレーション）

マイグレーションファイルは1本のみです。DDLの全文と詳細な解説は実ファイルを参照してください。

| ファイル | 内容 |
|----------|------|
| `src/main/resources/db/migration/V1__Create_memos_table.sql` | memosテーブル・インデックス2本・updated_at自動更新トリガー・コメントの作成 |

**実行タイミング**: アプリケーション起動時にFlywayが自動実行します（実行履歴は `flyway_schema_history` テーブルに記録され、適用済みのファイルは再実行されません）。

**注意（checksum）**: 適用済みのマイグレーションファイルを書き換えると、次回起動時に checksum mismatch エラーになります。学習環境では `docker compose down -v` でDBをリセットしてください。

### 5.5 PostgreSQL特有の機能（このプロジェクトで学べるもの）

- **BIGSERIAL型**: 自動採番。MySQLの `BIGINT AUTO_INCREMENT` に相当
- **トリガー**: `updated_at` の自動更新を実装（MySQLの `ON UPDATE CURRENT_TIMESTAMP` 相当）
- **GINインデックス + to_tsvector**: 全文検索の仕組み（現状は教材として作成のみ。§5.3参照）
- **pg_trgm拡張**（実装演習）: トライグラムによるLIKE部分一致の高速化

## 6. 画面設計

### 6.1 画面遷移図

```
        ┌──────────────┐
        │  メモ一覧    │ ← トップページ (/memos)
        │  (list)      │
        └──────────────┘
         │  │  │  │
         │  │  │  └── [削除] 確認ダイアログ → POST /memos/{id}/delete → 一覧へ
         │  │  │
         │  │  └───── [編集] ┌──────────┐
         │  │                │ メモ編集 │ → 更新後、詳細へ
         │  │                │ (edit)   │
         │  │                └──────────┘
         │  │
         │  └── [タイトル/詳細] ┌──────────┐
         │                      │ メモ詳細 │ →（編集・削除も可能）
         │                      │ (show)   │
         │                      └──────────┘
         │
         └── [新規作成] ┌──────────┐
                        │ メモ新規 │ → 保存後、一覧へ
                        │ (new)    │
                        └──────────┘
```

### 6.2 URL設計（ルーティング）

| 画面/処理 | HTTPメソッド | URL | 説明 |
|------|-------------|-----|------|
| メモ一覧 | GET | /memos | メモ一覧を表示 |
| メモ一覧（検索） | GET | /memos?keyword={keyword} | キーワード検索結果を表示 |
| メモ詳細 | GET | /memos/{id} | 指定IDのメモ詳細を表示 |
| メモ新規作成フォーム | GET | /memos/new | 新規作成フォームを表示 |
| メモ新規作成処理 | POST | /memos | メモを新規作成 |
| メモ編集フォーム | GET | /memos/{id}/edit | 編集フォームを表示 |
| メモ更新処理 | POST | /memos/{id}/update | メモを更新 |
| メモ削除処理 | POST | /memos/{id}/delete | メモを削除 |

**補足**: ルートパス（/）へのマッピングはありません。存在しないURLにアクセスすると404エラー画面が表示されます（§11参照）。

### 6.3 画面一覧とテンプレート

| 画面 | テンプレート | 主な表示要素 |
|------|--------------|--------------|
| メモ一覧 | `templates/memos/list.html` | 検索フォーム、新規作成ボタン、メモ一覧（タイトル・作成日時・本文100文字プレビュー・詳細/編集/削除ボタン）、フラッシュメッセージ |
| メモ詳細 | `templates/memos/show.html` | タイトル、本文（改行保持）、作成/更新日時、編集/削除/一覧に戻るボタン |
| メモ新規作成 | `templates/memos/new.html` | タイトル・本文の入力フォーム、バリデーションエラー表示 |
| メモ編集 | `templates/memos/edit.html` | 既存値入りフォーム、作成/更新日時（表示のみ）、バリデーションエラー表示 |
| エラー | `templates/error/error.html` | エラーメッセージ、トップに戻る/前に戻るボタン |
| 共通部品 | `templates/fragments/parts.html` | ヘッダー（ナビ、現在地ハイライト）・フッターのフラグメント定義 |

**レイアウト共通化**: 各画面のヘッダー・フッターは `th:replace="~{fragments/parts :: siteHeader('list')}"` のようにThymeleaf標準のフラグメント機能で共通化しています（Thymeleaf Layout Dialect は使用していません）。

**日時の表示形式**: `yyyy年MM月dd日 HH:mm`（例: 2026年07月21日 10:30）。フォーマットは `MemoDto.fromEntity()` がサーバー側で行い、画面には整形済み文字列を渡します。

### 6.4 レスポンシブデザイン

- デスクトップ（1024px以上）: 通常レイアウト
- タブレット・モバイル: 1カラムレイアウト（`static/css/style.css` のメディアクエリで制御）

## 9. クラス設計

実装の全文は各ソースファイル（詳細な学習コメント付き）を参照してください。ここでは**構造と設計判断**をまとめます。

### 9.1 クラス一覧と公開メソッド

#### Memo（entity）

| フィールド | 型 | 対応カラム |
|-----------|-----|-----------|
| id | Long | id |
| title | String | title |
| content | String | content |
| createdAt | LocalDateTime | created_at |
| updatedAt | LocalDateTime | updated_at |

スネークケース→キャメルケースの変換は `map-underscore-to-camel-case: true` で自動化。

#### MemoDto（dto）

| フィールド | 型 | バリデーション |
|-----------|-----|---------------|
| id | Long | なし（URLから取得するため） |
| title | String | @NotBlank、@Size(min=1, max=200) |
| content | String | @NotBlank |
| createdAtFormatted | String | なし（表示用。`yyyy年MM月dd日 HH:mm` 形式） |
| updatedAtFormatted | String | なし（表示用） |

変換メソッド: `fromEntity(Memo)`（Entity→DTO、日時フォーマット込み）/ `toEntity()`（DTO→Entity、日時はコピーしない）

**設計ポイント**: 日時を `LocalDateTime` ではなく**フォーマット済み文字列**で持つことで、テンプレート側のフォーマット処理を不要にしています。また、日時をフォームで往復させない（改ざん面を増やさない）設計です。

#### MemoMapper（mapper）

| メソッド | 戻り値 | 対応SQL（MemoMapper.xml） | 状態 |
|----------|--------|---------------------------|------|
| findAll() | List\<Memo\> | 全件取得（created_at降順） | 使用中 |
| findById(Long) | Optional\<Memo\> | ID指定1件取得 | 使用中 |
| insert(Memo) | void | INSERT（useGeneratedKeysでID採番） | 使用中 |
| update(Memo) | int | UPDATE（updated_atはトリガー任せ） | 使用中 |
| deleteById(Long) | int | DELETE | 使用中 |
| searchByKeyword(String) | List\<Memo\> | LIKE部分一致検索 | 使用中 |
| findAllWithPaging(int, int) | List\<Memo\> | LIMIT/OFFSET | **未使用（実装演習用）** |
| count() | long | COUNT(*) | **未使用（実装演習用）** |

**設計ポイント**: `findById` は `Optional<Memo>` を返し、null安全に「存在しないかもしれない」ことを型で表現しています。

#### MemoService（service）

| メソッド | トランザクション | 内容 |
|----------|-----------------|------|
| findAll() | readOnly | 全件取得しDTOに変換 |
| findById(Long) | readOnly | 1件取得。無ければ MemoNotFoundException |
| create(MemoDto) | 書き込み | DTO→Entity変換して登録 |
| update(MemoDto) | 書き込み | 存在チェック→更新→更新件数のダブルチェック |
| delete(Long) | 書き込み | 存在チェック→削除 |
| search(String) | readOnly | キーワード検索。**空・nullなら全件取得にフォールバック** |
| findAllWithPaging(int, int) | readOnly | ページング取得（**未使用・実装演習用**） |
| count() / getTotalPages(int) | readOnly | 総件数・総ページ数（**未使用・実装演習用**） |

**設計ポイント**:
- **インターフェース + Impl の分離は採用していません**。実装クラスが1つしかない学習用アプリでは、`MemoService` インターフェース + `MemoServiceImpl` の分離は間接参照が増えるだけでメリットが薄いため、具象クラス1つの構成にしています（実務では複数実装・モック差し替え・AOP等の理由で分離する場合があります）
- 「キーワードが空なら全件」という判断はServiceが持ち、Controllerは分岐せず `search(keyword)` を呼ぶだけにしています
- 参照系メソッドには `@Transactional(readOnly = true)` を付与しています（効果と自己呼び出しの落とし穴はソースコメント参照）

#### MemoController（controller）

| メソッド | ルーティング | 内容 |
|----------|-------------|------|
| list | GET /memos | 一覧・検索（`search()` に委譲） |
| newMemo | GET /memos/new | 新規作成フォーム表示 |
| create | POST /memos | 登録。成功時は一覧へリダイレクト（PRG） |
| show | GET /memos/{id} | 詳細表示 |
| edit | GET /memos/{id}/edit | 編集フォーム表示 |
| update | POST /memos/{id}/update | 更新。エラー時はIDと日時を再設定して編集画面へ |
| delete | POST /memos/{id}/delete | 削除。一覧へリダイレクト |

**設計ポイント**:
- IDや日時はフォームの hidden で送らず、URLパス（@PathVariable）とDB再取得で復元します（改ざん対策）
- 削除・更新はGETではなくPOSTを使用（GETは安全な操作に限定する）

### 9.2 例外設計

| 例外 | ハンドラ | HTTPステータス | ログレベル |
|------|----------|---------------|-----------|
| MemoNotFoundException | handleMemoNotFoundException | 404 | WARN |
| NoResourceFoundException（未定義URL） | handleNoResourceFoundException | 404 | WARN |
| IllegalArgumentException | handleIllegalArgumentException | 400 | ERROR |
| Exception（フォールバック） | handleException | 500 | ERROR |

すべて `templates/error/error.html` を表示します。

**学習ポイント（重要な落とし穴）**: `@ExceptionHandler` がビューを返す場合、例外クラス側の `@ResponseStatus` は適用されず、**ハンドラメソッド側に `@ResponseStatus` を明示しないとステータスが200になります**。詳細は `GlobalExceptionHandler.java` のコメントを参照してください。

### 13.1 初心者向け学習の進め方

1. **まずは動かしてみる**: 環境構築を完了させ、各機能を操作して動作を確認
2. **設計書（本書）で全体像を掴む**: 3層の役割・URL設計・DB設計
3. **コードを読む**: Controller → Service → Mapper の流れを追う。ソースの学習コメントが「なぜ」を説明しています
4. **SQLを確認する**: `MemoMapper.xml` のSQLを読み、psqlで直接実行してみる
5. **小さな変更を加える**: メッセージ文言・CSS・バリデーションルールの変更
6. **実装演習に挑戦**: §12.2の演習（ページング等）
7. **テストを書く**: 既存の30テストを参考に、変更した箇所のテストを追加

### 13.2 つまづきやすいポイント

| 症状 | 確認ポイント |
|------|-------------|
| PostgreSQLに接続できない | コンテナ起動状態（`docker compose ps`）、application.ymlの接続情報 |
| Flyway checksum mismatch | 適用済みマイグレーションを書き換えた場合に発生。`docker compose down -v` でリセット |

## 付録

### A. 用語集

- **CRUD**: Create（作成）、Read（読み取り）、Update（更新）、Delete（削除）の頭文字
- **MVC**: Model-View-Controller。アプリケーションを3つの役割に分離する設計パターン
- **DTO**: Data Transfer Object。層間でデータを受け渡すためのオブジェクト
- **Entity**: データベーステーブルに対応するJavaオブジェクト
- **Mapper**: MyBatisのデータアクセスインターフェース
- **2waySQL**: 動的SQLとしても静的SQLとしても利用できるSQL記述方式
- **DI**: Dependency Injection（依存性注入）
- **PRGパターン**: Post-Redirect-Get。二重送信を防ぐリダイレクト設計
- **フラグメント**: Thymeleafのテンプレート部品化機能（th:fragment / th:replace）
- **スライステスト**: アプリの一部の層だけを起動するテスト（@WebMvcTest など）

### B. トラブルシューティング

#### ポート8080が既に使用されている
```bash
# Windowsの場合
netstat -ano | findstr :8080
taskkill /PID <PID> /F

# macOS/Linuxの場合
lsof -i :8080
kill -9 <PID>
```

#### PostgreSQL接続エラー
```bash
# コンテナが起動しているか確認
docker compose ps
docker compose logs postgres
```

#### Flywayマイグレーションエラー（checksum mismatch など）
```bash
# 学習環境ではDBリセットが最も確実（データは消えます）
docker compose down -v
docker compose up --build
```
※ 本プロジェクトには flyway-maven-plugin を入れていないため、`mvn flyway:migrate` 等のMavenコマンドは**使えません**。マイグレーションはアプリ起動時に自動実行されます。

### C. 開発時の便利なコマンド

```bash
# すべてDockerで起動（推奨）
docker compose up --build

# DBのみ起動してアプリはローカル実行
docker compose up -d postgres
mvn spring-boot:run

# テスト実行
mvn test

# JAR作成
mvn clean package

# PostgreSQLに接続（psql）
docker compose exec postgres psql -U memoapp_user -d memoapp
```
