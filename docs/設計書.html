第 1 章

システム概要

1.1 目的

本アプリケーションは、開発未経験者が Web アプリケーション開発の基礎を学ぶための教材として、 基本的な CRUD 操作を実装したメモ帳アプリケーションです。 従来型のサーバーサイドレンダリング（MVC）構成を採用し、3 層レイヤーアーキテクチャで実装しています。

実務を意識した技術選定:

  • PostgreSQL: エンタープライズ開発で広く採用されている RDBMS
  • MyBatis + 2waySQL: 日本の業務システム開発で標準的に使われる技術
  • Flyway: データベースマイグレーション管理（実務必須スキル）

1.2 対象ユーザー

  • Web アプリケーション開発の学習を始めたばかりの初心者
  • 従来型の MVC アーキテクチャを学びたい開発者
  • Java / Spring Boot の基礎を学びたい開発者
  • 実務で使われる 2waySQL（MyBatis）を習得したい開発者

1.3 主な機能

機能状態
メモの新規作成実装済み
メモの一覧表示実装済み
メモの詳細表示実装済み
メモの編集実装済み
メモの削除（確認ダイアログあり）実装済み
メモの検索(キーワード部分一致)実装済み
ページング未実装（Service/Mapper の部品のみ用意。実装演習）

第 2 章

アーキテクチャ設計

2.1 3 層レイヤーアーキテクチャ

┌─────────────────────────────────────┐ │ プレゼンテーション層 │ │ (Presentation Layer) │ │ - Controller │ │ - Thymeleaf (View) │ │ - JavaScript (Vanilla) │ └─────────────────────────────────────┘ ↓↑ ┌─────────────────────────────────────┐ │ ビジネスロジック層 │ │ (Business Logic Layer) │ │ - Service │ │ - DTO (Data Transfer Object) │ └─────────────────────────────────────┘ ↓↑ ┌─────────────────────────────────────┐ │ データアクセス層 │ │ (Data Access Layer) │ │ - Mapper (MyBatis) │ │ - SQL (2waySQL in XML) │ │ - Entity │ │ - Database (PostgreSQL) │ └─────────────────────────────────────┘

2.1.1 プレゼンテーション層

責務: ユーザーとのインタラクション、画面表示、入力値の受け取り。

  • Controller: HTTP リクエストを受け取り、Service を呼び出し、View に渡すデータを準備
  • Thymeleaf: HTML を生成するテンプレートエンジン。共通部品は th:fragment / th:replace で共通化
  • JavaScript (Vanilla): フレームワークや jQuery を使わない素の JavaScript。削除確認ダイアログ・クライアントサイドバリデーション・フラッシュメッセージの自動非表示を実装

2.1.2 ビジネスロジック層

責務: ビジネスルールの実装、トランザクション管理、データの加工。

  • Service: ビジネスロジックを実装するクラス
  • DTO: 層間でデータを受け渡すためのオブジェクト

2.1.3 データアクセス層

責務: データベースへのアクセス、永続化処理。

  • Mapper: MyBatis の Mapper インターフェース（データベース操作の定義）
  • SQL: XML ファイルに記述された 2waySQL
  • Entity: データベーステーブルに対応する Java オブジェクト
  • Database: PostgreSQL（実際のデータ格納先）

2.2 技術選定の理由

なぜ PostgreSQL か？

  1. エンタープライズでの採用実績: 金融・公共システムなどで広く使われている
  2. 機能の豊富さ: トランザクション、JSON 型、全文検索など
  3. 標準 SQL 準拠: SQL 標準に忠実で移植性が高い
  4. オープンソース: 商用利用も可能でライセンスコストがかからない

なぜ MyBatis + 2waySQL か？

  1. 実務での標準: 日本の業務システム開発で広く採用されている
  2. SQL の可視性: SQL を直接記述するため、パフォーマンスチューニングが容易
  3. 複雑なクエリへの対応: ORM では表現しにくい複雑な JOIN やサブクエリも自由に記述
  4. 2waySQL: 動的な SQL とテスト可能な静的 SQL を両立

なぜ Flyway によるマイグレーション管理か？

  1. SQL の学習機会: マイグレーションファイルを手書きするため、SQL 学習が阻害されない
  2. バージョン管理: データベーススキーマの変更履歴を管理できる
  3. チーム開発: 開発者間でデータベース構造を同期できる
  4. 実務スキル: 実際のプロジェクトで必須の技術
重要: ddl-auto: update のような完全自動生成は使用せず、マイグレーションファイルを手書きします。マイグレーションはアプリケーション起動時に自動実行されます（Maven コマンドによる手動実行は本プロジェクトでは構成していません。詳細は §10 設定ファイルを参照）。

なぜサーバーサイドレンダリング（MVC）か？

  1. シンプルさ: クライアント側の複雑な状態管理が不要
  2. 学習曲線: フロントエンドフレームワークの習得が不要
  3. 従来型の理解: モダン構成（learn-memo-app-modern）との比較学習のベースライン

3.2 機能詳細仕様

F001: メモ一覧表示

  • 入力: なし
  • 処理: (1) データベースからすべてのメモを取得 (2) 作成日時の降順（新しい順）でソート (3) 一覧画面に表示（本文は 100 文字で省略表示）
  • 出力: メモ一覧（タイトル、作成日時、本文プレビュー）
  • 例外処理: データベース接続エラー時はエラーメッセージを表示（500）

F002: メモ詳細表示

  • 入力: メモ ID（URL パス）
  • 処理: (1) 指定された ID のメモをデータベースから取得 (2) 詳細画面に表示
  • 出力: メモ詳細（タイトル、内容、作成日時、更新日時）
  • 例外処理: メモが存在しない場合は 404 エラー画面を表示

F003: メモ新規作成

  • 入力: タイトル、本文
  • 処理: (1) 入力値のバリデーション (2) データベースに保存（作成日時・更新日時は DB 側の DEFAULT で自動設定） (3) 一覧画面にリダイレクト（PRG パターン）
  • 出力: 作成成功メッセージ（フラッシュメッセージ）、一覧画面への遷移
  • バリデーション: タイトル = 必須、1〜200 文字（DB の VARCHAR(200) に対応）／ 本文 = 必須、文字数制限なし（DB の TEXT 型に対応）
  • 例外処理: バリデーションエラー時は入力値を保持して再表示

3.2 機能詳細仕様（続き）

F004: メモ編集

  • 入力: メモ ID（URL パス）、タイトル、本文
  • 処理: (1) 指定された ID のメモをデータベースから取得（存在チェック） (2) 入力値のバリデーション (3) データベースを更新（更新日時は DB トリガーで自動更新） (4) 詳細画面にリダイレクト
  • 出力: 更新成功メッセージ、詳細画面への遷移
  • バリデーション: F003 と同じ
  • 例外処理: メモが存在しない場合は 404 エラー。バリデーションエラー時は入力値を保持して再表示（作成・更新日時はサーバー側で再取得して表示を維持）

F005: メモ削除

  • 入力: メモ ID（URL パス）
  • 処理: (1) JavaScript（script.js の initDeleteConfirmation()）で削除確認ダイアログを表示 (2) ユーザーが OK を押した場合のみ削除リクエストを送信 (3) 指定された ID のメモをデータベースから削除 (4) 一覧画面にリダイレクト
  • 出力: 削除成功メッセージ、一覧画面への遷移
  • 例外処理: メモが存在しない場合は 404 エラー

F006: メモ検索

  • 入力: 検索キーワード（クエリパラメータ keyword）
  • 処理: (1) タイトルまたは本文にキーワードを含むメモを LIKE 部分一致で取得 (2) キーワードが空・未指定の場合は全件取得（判断は Service 層が担う） (3) 一覧画面に検索結果を表示
  • 出力: 検索結果のメモ一覧
  • 例外処理: 検索結果が 0 件の場合は「メモがありません」を表示
  • 既知の制限: LIKE のワイルドカード文字（% _）はエスケープしていない（詳細は MemoMapper.xml のコメント参照）

非機能要件

4.1 パフォーマンス要件

  • 一覧画面の初期表示: 3 秒以内
  • CRUD 操作のレスポンス: 1 秒以内
  • 同時アクセス: 10 ユーザーまで（学習用のため）

4.2 可用性要件

  • 学習用のため、高可用性は求めない
  • ローカル環境での動作を前提

4.3 セキュリティ要件

  • XSS 対策: Thymeleaf の自動エスケープ機能を使用
  • SQL インジェクション対策: MyBatis のプリペアドステートメント（#{} 記法）を使用
  • CSRF 対策: 未実装（Spring Security 導入時の学習課題）
  • 認証・認可: 未実装（将来の拡張として）

4.4 保守性要件

  • コードの可読性: 初心者が理解できるシンプルなコード
  • コメント: 教育目的の詳細な日本語コメント（Why・仕組み・落とし穴を中心に記載）
  • ログ出力: エラー発生時の原因特定が容易なログ設計（404 系は WARN、システム異常は ERROR）
  • SQL の可視性: 2waySQL により SQL が明示的で保守しやすい

4.5 移植性要件

  • OS: Windows、macOS、Linux で動作
  • データベース: PostgreSQL 16（docker-compose.yml は postgres:16-alpine を使用）
  • Java: Java 17 以上
  • ブラウザ: Chrome、Firefox、Edge、Safari（最新版）

5.3 インデックス設計

インデックス名カラム種類目的

重要（学習ポイント）: idx_memos_fulltext は全文検索演算子（@@）専用のインデックスであり、現在の検索機能（LIKE 部分一致）からは使われません。LIKE 検索を高速化するには pg_trgm 拡張 + gin_trgm_ops のインデックスが必要です（実装演習。V1__Create_memos_table.sql のコメント参照）。

5.4 DDL（Flyway マイグレーション）・5.5 PostgreSQL 特有の機能

マイグレーションファイルは 1 本のみです（src/main/resources/db/migration/V1__Create_memos_table.sql）。内容は memos テーブル・インデックス 2 本・updated_at 自動更新トリガー・コメントの作成です。DDL の全文と詳細な解説は実ファイルを参照してください。実行タイミング: アプリケーション起動時に Flyway が自動実行します（実行履歴は flyway_schema_history テーブルに記録され、適用済みのファイルは再実行されません）。

注意（checksum）: 適用済みのマイグレーションファイルを書き換えると、次回起動時に checksum mismatch エラーになります。学習環境では docker compose down -v で DB をリセットしてください。

PostgreSQL 特有の機能（このプロジェクトで学べるもの）: BIGSERIAL 型（自動採番。MySQL の BIGINT AUTO_INCREMENT 相当）／ トリガー（updated_at の自動更新。MySQL の ON UPDATE CURRENT_TIMESTAMP 相当）／ GIN インデックス + to_tsvector（全文検索の仕組み。現状は教材として作成のみ）／ pg_trgm 拡張（実装演習。トライグラムによる LIKE 部分一致の高速化）。

MemoService（service）

メソッドトランザクション内容

付録

A. 用語集

  • CRUD: Create（作成）、Read（読み取り）、Update（更新）、Delete（削除）の頭文字
  • MVC: Model-View-Controller。アプリケーションを 3 つの役割に分離する設計パターン
  • DTO: Data Transfer Object。層間でデータを受け渡すためのオブジェクト
  • Entity: データベーステーブルに対応する Java オブジェクト
  • Mapper: MyBatis のデータアクセスインターフェース
  • 2waySQL: 動的 SQL としても静的 SQL としても利用できる SQL 記述方式
  • DI: Dependency Injection（依存性注入）
  • PRG パターン: Post-Redirect-Get。二重送信を防ぐリダイレクト設計
  • フラグメント: Thymeleaf のテンプレート部品化機能（th:fragment / th:replace）
  • スライステスト: アプリの一部の層だけを起動するテスト（@WebMvcTest など）
