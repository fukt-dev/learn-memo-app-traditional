# 例外処理とエラーページ

`GlobalExceptionHandler.java` / `MemoNotFoundException.java` / `error.html` が、
エラー発生時に何をしているのか、その背景をまとめます。

## この記事で分かること

- 例外ハンドラがないと何が起きるか(Whitelabel Error Page)
- `@ControllerAdvice` と `@ExceptionHandler` による一括処理
- 用途に特化したカスタム例外を作る理由
- `@ResponseStatus` の落とし穴(例外クラス側だけでは効かない)
- checked 例外と unchecked 例外の違い、HTTP ステータスの使い分け

## 例外ハンドラがない世界

エラーを何も処理しないと、Spring Boot は既定の「Whitelabel Error Page」を表示します。

```
Whitelabel Error Page
There was an unexpected error (type=Internal Server Error, status=500).
```

ユーザーには何が起きたか分からず、次にどうすればよいかも示されません。
そこで例外を一箇所で捕まえ、分かりやすいエラー画面と適切な HTTP ステータスに
変換するのが例外ハンドラの役割です。

## `@ControllerAdvice` と `@ExceptionHandler`

`@ControllerAdvice` を付けたクラスは、**全 Controller で発生した例外** を横断的に捕捉できます。
その中で `@ExceptionHandler(例外クラス)` を付けたメソッドが、対応する例外を処理します。

```java
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MemoNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)      // ← 404 を明示(後述)
    public String handleMemoNotFound(MemoNotFoundException ex, Model model) {
        model.addAttribute("errorMessage", ex.getMessage());
        return "error/error";                  // 共通エラー画面を返す
    }
}
```

各 Controller メソッドに `try-catch` を書き散らす代わりに、ここに集約することで
「エラー画面とログ出力を 1 か所で管理できる」のが利点です。

### ハンドラの優先順位とフォールバック

Spring は「発生した例外に最も型が近いハンドラ」を選びます。このアプリは
具体的な例外から順に、次の 4 つを用意しています。

| ハンドラ対象 | 意味 | ステータス | ログレベル |
|--------------|------|:----------:|:----------:|
| `MemoNotFoundException` | メモが存在しない | 404 | WARN |
| `NoResourceFoundException` | 未マッピングの URL | 404 | WARN |
| `IllegalArgumentException` | 不正な引数 | 400 | ERROR |
| `Exception`(フォールバック) | 予期しない例外全般 | 500 | ERROR |

最後の `Exception` は「どれにも当てはまらなかった例外」を受ける最後の砦です。
逆に言うと、`NoResourceFoundException` の専用ハンドラを用意しないと、
ただの URL タイプミスまでこのフォールバックに吸い込まれ、「システムエラー画面 +
ERROR ログ」になってしまいます。4xx はユーザー起因のよくある事象なので WARN、
5xx はサーバーの異常なので ERROR、と分けるのが目安です。

## カスタム例外を作る理由

`MemoNotFoundException` のような専用の例外クラスを定義すると、次の利点があります。

- **意図が型で伝わる**: 「不正な引数」ではなく「メモが見つからない」が一目で分かる
- **ステータスを対応づけられる**: この例外は 404、という取り決めを型で表現できる
- **ハンドリングを分けられる**: 上表のように、例外ごとに画面・ログ・ステータスを変えられる

`MemoService` は取得失敗時に `orElseThrow` でこの例外を投げます。`findById` が返す
`Optional`(値が有るか無いかのコンテナ)に対し「無ければ例外」を宣言する書き方です。

```java
return memoMapper.findById(id)
        .map(MemoDto::fromEntity)
        .orElseThrow(() -> new MemoNotFoundException("メモが見つかりません: id=" + id));
```

## 落とし穴: `@ResponseStatus` は「捕捉されない」場合だけ効く

`MemoNotFoundException` クラスには `@ResponseStatus(HttpStatus.NOT_FOUND)` が
付いています。ところがこれが効くのは、例外が **どの `@ExceptionHandler` にも
捕捉されず** に Spring の既定処理まで届いた場合だけです。

このアプリのようにハンドラが例外を捕捉してビューを返すと、Spring から見れば
「正常にレスポンスを生成できた」ことになり、何も指定しなければステータスは
**200(OK)** になってしまいます。エラー画面を見せているのにステータスは 200、
という機械には分からない状態です。

そのため、ハンドラメソッド側にも `@ResponseStatus(HttpStatus.NOT_FOUND)` を
明示的に付けて 404 を確定させています。クラス側の指定は「この例外は 404 を意図する」
という仕様の宣言と、将来ハンドラを外したときの保険として残しています。

## checked 例外と unchecked 例外

`MemoNotFoundException` は `RuntimeException` を継承した **unchecked 例外** です。

| 種類 | 継承元 | `try-catch` | 例 |
|------|--------|-------------|-----|
| checked 例外 | `Exception` | 呼び出し側で必須 | `IOException` |
| unchecked 例外 | `RuntimeException` | 任意 | `NullPointerException` |

ビジネスロジックの例外を unchecked にすると、呼び出し側が毎回 `try-catch` を
書かずに済み、`@ControllerAdvice` で一括処理できます。なお `@Transactional` の
自動ロールバックが既定で効くのも unchecked 例外です(詳細は
[トランザクション.md](トランザクション.md))。

## モダン版との対比

モダン版(`learn-memo-app-modern`)も同じ発想の例外ハンドラを持ちますが、
返すものが「エラー画面」ではなく「エラー JSON」です。

| 観点 | 従来型(この記事) | モダン版(REST API) |
|------|-------------------|---------------------|
| ハンドラ | `@ControllerAdvice` | `@RestControllerAdvice` |
| 返すもの | エラー画面のビュー名(`error/error`) | エラー内容の JSON |
| 見せ方 | サーバーが HTML を再描画 | React が JSON を受けて表示 |

「例外を一箇所に集約し、種類ごとにステータスを割り当てる」骨格は共通で、
出力が HTML か JSON かが両者の違いです。
