# テスト(JUnit 5 + Mockito + MockMvc)

`MemoServiceTest.java` / `MemoControllerTest.java` / `MemoDtoTest.java` が、
どんな道具で何を検証しているのか、その背景をまとめます。

## この記事で分かること

- 学習リポジトリでテストを書く意味
- 単体テスト・Web レイヤーテスト・純ロジックテストの役割分担
- Mockito の基本(`@Mock` / `@InjectMocks` / `when` / `verify` / `ArgumentCaptor`)
- `@WebMvcTest` と MockMvc による擬似 HTTP リクエスト
- テストの読み方(Given-When-Then)と、`@MockBean` 廃止の落とし穴

## なぜテストを書くのか

このリポジトリは学習用ですが、テストは「動く見本」として二重に役立ちます。

- **仕様の記録**: 「空タイトルは入力画面に戻す」「存在しない ID は 404」といった
  仕様が、日本語の `@DisplayName` 付きで実行可能な形で残ります
- **退行の検出**: コードを直したとき、壊れたらすぐ赤くなります。手で全画面を
  操作し直す必要がありません

このアプリのテストは 3 本あり、それぞれ対象とする「層」が違います。

| ファイル | 対象 | 道具 | DB |
|----------|------|------|:--:|
| `MemoServiceTest.java` | Service のロジック | JUnit 5 + Mockito | 不要 |
| `MemoControllerTest.java` | Controller の HTTP 入出力 | `@WebMvcTest` + MockMvc | 不要 |
| `MemoDtoTest.java` | DTO ⇔ Entity の変換 | 素の JUnit 5 | 不要 |

いずれも本物の DB を使いません。「隣の層を偽物に差し替え、自分の層だけを切り出して
調べる」のが共通の発想で、それを可能にしているのがコンストラクタ注入の設計です
([DIとLombok.md](DIとLombok.md))。

## 単体テスト: Service と Mockito

Service は Mapper に依存します。テストでは本物の Mapper の代わりに **モック
(指示した通りに振る舞う偽物)** を注入し、Service のロジックだけを試します。

```java
@ExtendWith(MockitoExtension.class)   // JUnit 5 に Mockito を組み込む
class MemoServiceTest {
    @Mock        private MemoMapper memoMapper;   // 偽物の Mapper
    @InjectMocks private MemoService memoService; // 偽物を注入した本物の Service
}
```

`@Mock` がモックを作り、`@InjectMocks` がそれをコンストラクタ経由で Service に
注入します(コンストラクタ注入だからこそ、Mockito が差し替えられます)。
Mockito の基本は「振る舞いを仕込む」「呼ばれ方を検証する」の 2 つです。

| 用途 | 書き方 | 意味 |
|------|--------|------|
| 戻り値を仕込む | `when(mock.method()).thenReturn(値)` | 呼ばれたらこの値を返す |
| 呼ばれたか検証 | `verify(mock).method()` | 呼ばれたか |
| 呼ばれないか検証 | `verify(mock, never()).method()` | 一度も呼ばれていないか |
| 引数を捕まえる | `ArgumentCaptor` | 渡された引数の中身を検証する |

たとえば「存在しない ID なら例外を投げ、更新は実行しない」ことはこう書けます。

```java
when(memoMapper.findById(999L)).thenReturn(Optional.empty());   // 「無い」を仕込む
assertThatThrownBy(() -> memoService.update(dto))
        .isInstanceOf(MemoNotFoundException.class);
verify(memoMapper, never()).update(any());                      // 更新されないことを確認
```

戻り値が `void` の `insert()` は戻り値で検証できないため、`ArgumentCaptor` で
「Service が DTO→Entity 変換をして Mapper に渡した引数」を捕まえて中身を確認します。

## Web レイヤーテスト: MockMvc

Controller のテストは、実際の HTTP サーバーを立てずに MockMvc がリクエストを
模擬します。`@WebMvcTest(MemoController.class)` は **Controller と例外ハンドラだけ** を
読み込み、Service や Mapper は読み込みません(だから Service は偽物を用意します)。
軽くて速いのが利点です。

```java
mockMvc.perform(get("/memos/999"))            // 擬似 GET リクエスト
        .andExpect(status().isNotFound())     // 404 か
        .andExpect(view().name("error/error"));
```

`perform` でリクエストを送り、`andExpect` でステータス・ビュー名・モデルを検証します。
このテストは「存在しない ID は 404」「タイトルが空なら入力画面に戻り登録しない」
「未マッピング URL は 404」といった、[例外処理とエラーページ.md](例外処理とエラーページ.md)や
[Bean-Validation.md](Bean-Validation.md)の取り決めが崩れていないかを守るガードです。

### 落とし穴: `@MockBean` は廃止予定

Web レイヤーテストでは Service を偽物に差し替えます。かつては `@MockBean` を
使いましたが、Spring Boot 3.4 以降で非推奨となり、Spring Framework 本体が提供する
`@MockitoBean`(パッケージ `org.springframework.test.context.bean.override.mockito`)に
置き換わりました。このアプリの `MemoControllerTest` は新しい `@MockitoBean` を
使っています。古い記事のサンプルをそのままコピーすると非推奨警告が出るので注意します。

## 純ロジックテスト: 素の JUnit

`MemoDtoTest` は `fromEntity` / `toEntity` の変換だけを試します。この変換は
フレームワークに依存しない純粋な Java ロジックなので、Mockito も Spring も使わず
素の JUnit で書きます。日時が `yyyy年MM月dd日 HH:mm` 形式になること、日時が
`null` でも落ちないこと、DTO→Entity で日時をコピーしないことなどを固定しています。
依存が少ないテストほど速く壊れにくい、というテストピラミッドの土台の実例です。

## テストの読み方: Given-When-Then

各テストは 3 段構成です(Arrange / Act / Assert とも呼びます)。

| 段 | やること |
|----|----------|
| Given(前提) | テストデータとモックの振る舞いを用意 |
| When(操作) | テスト対象のメソッドを 1 回呼ぶ |
| Then(検証) | 戻り値・例外・呼ばれ方を確かめる |

`@DisplayName` にその内容を日本語で書くことで、テスト結果のレポートが「仕様書」の
ように読めます。テストを読むときは、まず `@DisplayName` で意図をつかむと迷いません。

## 実行方法

```bash
mvn test                                 # 全テスト
mvn test -Dtest=MemoControllerTest       # クラスを絞る
```

## モダン版との対比

モダン版(`learn-memo-app-modern`)**も同じ 3 層構成のテスト**(Service の単体テストと
`@WebMvcTest` の Web レイヤーテスト)を備えています。骨格は共通で、細部が異なります。

| 観点 | 従来型(この記事) | モダン版 |
|------|-------------------|----------|
| 言語 | Java + JUnit 5 | Kotlin + JUnit 5 |
| Web 層で検証するもの | ステータス・ビュー名・モデル | ステータス・JSON(`jsonPath`) |
| モックライブラリの補助 | 追加不要 | `mockito-kotlin`(`any()` の null 問題対策) |

従来型はサーバーが返す HTML(ビュー名)を、モダン版は返す JSON を検証する、という
違いは、そのまま「画面を返すか JSON を返すか」というアーキテクチャの差の反映です。
