# トランザクション

`MemoService.java` の各メソッドに付いた `@Transactional` が何をしているのか、
その背景をまとめます。

## この記事で分かること

- トランザクションとは何か、なぜ必要か
- `@Transactional` を付けるとメソッドの前後で何が起きるか
- `readOnly = true` の効果
- どんな例外でロールバックされるのか(unchecked と checked の違い)
- 同じクラス内の呼び出しでは `@Transactional` が効かない落とし穴

## トランザクションとは

**トランザクション** は、複数の DB 操作を「全部成功」か「全部失敗」のどちらかに
まとめる仕組みです。有名なたとえは銀行の振込で、「引き落としは成功したが
振込は失敗」という中途半端な状態を防ぎます。途中で失敗したら、それまでの操作を
なかったことにする(ロールバック)ことで、データの整合性を守ります。

## `@Transactional` の動作

Spring では、メソッドに `@Transactional` を付けるだけでそのメソッドが
トランザクションの境界になります。開発者が「開始」「コミット」「ロールバック」を
手書きする必要はありません。

```java
@Transactional
public void update(MemoDto dto) {
    findById(dto.getId());          // 存在チェック(なければ例外)
    int updatedCount = memoMapper.update(dto.toEntity());
    if (updatedCount == 0) {
        throw new MemoNotFoundException("メモが見つかりません: id=" + dto.getId());
    }
}
```

- **メソッド開始時**: トランザクションを開始する
- **正常終了時**: コミットする(変更が DB に確定する)
- **例外発生時**: ロールバックする(変更が取り消される)

このアプリでは、`create` / `update` / `delete` のようにデータを変更するメソッドに
`@Transactional` を付けています。

## `readOnly = true` の効果

参照だけのメソッド(`findAll` / `findById` / `search` など)には
`@Transactional(readOnly = true)` を付けています。これには 3 つの意味があります。

- PostgreSQL のトランザクションが READ ONLY モードになり、うっかり更新 SQL を
  実行すると検出できる
- ドライバやコネクションプールに「読み取り専用」と伝わり、最適化の余地が生まれる
- コードを読む人に「このメソッドはデータを変更しない」と一目で伝わる

## ロールバックの条件

見落としやすい点として、`@Transactional` が **自動でロールバックするのは
unchecked 例外(`RuntimeException` とそのサブクラス)だけ** です。
`Exception` を継承した checked 例外では、既定ではロールバックされません。

このアプリの `MemoNotFoundException` は `RuntimeException` を継承した unchecked 例外
(詳細は [例外処理とエラーページ.md](例外処理とエラーページ.md))なので、更新中にこれが
投げられれば、それまでの変更はきちんと取り消されます。

## 落とし穴: 自己呼び出しではプロキシを経由しない

`@Transactional` は、Spring が対象クラスを **プロキシ(代理オブジェクト)** で包み、
その代理がトランザクションの開始・終了を差し込むことで実現しています。
ここに落とし穴があります。**同じクラス内のメソッドを直接呼ぶと、この代理を
経由しない** ため、呼ばれた側の `@Transactional` が効きません。

```java
@Transactional(readOnly = true)
public int getTotalPages(int pageSize) {
    long totalCount = count();   // 同じクラスの count() を直接呼ぶ
    return (int) Math.ceil((double) totalCount / pageSize);
}

@Transactional(readOnly = true)
public long count() { ... }      // この場合、ここの指定は効かない
```

`getTotalPages()` から `count()` を内部呼び出しすると、`count()` 側の
アノテーションは無視されます。だから「外から呼ばれる入口」である
`getTotalPages()` 自身にも `@Transactional` を付けておく必要があります。
別クラスに切り出すか、入口側に付けるのが対処です。

## モダン版との対比

モダン版(`learn-memo-app-modern`)も **まったく同じ `@Transactional`** を使います。
これは MyBatis / JPA といった DB アクセス技術に依存しない、Spring 共通の機能だからです。

| 観点 | 従来型(この記事) | モダン版 |
|------|-------------------|----------|
| アノテーション | `@Transactional` | `@Transactional`(同じ) |
| 参照メソッド | `readOnly = true`(同じ) | `readOnly = true`(同じ) |
| 更新の反映 | Mapper の SQL がコミット時に確定 | JPA の変更がコミット時に確定 |

トランザクションの考え方は両アプリで共通です。違いは「その中で実行される
DB アクセスの書き方」(MyBatis の SQL か、JPA の操作か)だけです。
