# 学習用Webアプリケーション開発ロードマップ（たたき台）

Java (Spring / TERASOLUNA) + Javascript を軸に、「従来構成」と「モダン構成」の比較学習を通じて、実践的なスキル習得を目指すためのロードマップ案です。

---

## ステップ1：メモ帳アプリ（基本のCRUD）

まずは、Webアプリケーションの基本であるCRUD（作成・読み取り・更新・削除）を網羅したシンプルなアプリケーションを作成し、両構成の根本的な違いを体感します。

### テーマ
- 基本的なCRUD
- SPAとMVCの比較

### 学習目的
- サーバーサイドレンダリング (MVC) とクライアントサイドレンダリング (SPA) の違いを理解する
- モノリシックな構成と、API分離型（Headless）構成の違いを理解する
- 両構成におけるデータ処理の流れを掴む

### 作成するバージョン

#### v1 (従来型)
- **スタック**: Java (Spring MVC / TERASOLUNA) + JSP (または Thymeleaf) + Javascript (jQueryなど)
- **特徴**: サーバー側で画面を生成し、画面遷移もサーバーが制御する構成

#### v2 (モダン型)
- **スタック**: Java (Spring Boot) (REST APIのみ) + React (または Vue.js)
- **特徴**: サーバーはJSONを返すことに専念し、画面描画や遷移はすべてフロントエンド(React)が担当するSPA構成

---

## ステップ2：タスク管理アプリ（リレーションの導入）

メモ帳アプリ（単一の概念）から一歩進め、「1対多」のデータ関係（リレーション）を扱います。これにより、より実務に近いデータ構造の扱いに慣れます。

### テーマ
- 「1対多」のリレーション
- 状態管理

### 学習目的
- 「プロジェクト」と「タスク」のような、親（1）と子（多）の関係を持つデータの扱い方を学ぶ
- （モダン型の場合）フロントエンド側で、関連する複数のデータをどのように保持し、表示に反映させるか（状態管理）を学ぶ

### ステップ1からの難易度上昇ポイント
- データモデルが単一ではなくなる（例：ProjectテーブルとTaskテーブル）
- UIが複雑になる（例：「プロジェクトAに紐づくタスク一覧」の表示）

### 補足
このステップは、v2 (モダン型) のみを作成し、フロントエンドでの状態管理の複雑さに焦点を当てる形でも良いかもしれません。

---

## ステップ3：伝票処理アプリ（最終目標）

エンタープライズ開発（業務システム）の縮図とも言える、複雑なデータモデルとビジネスロジックを扱うアプリケーションに挑戦します。

### テーマ
- 複雑なデータモデル（多対多）
- 動的なUI
- ビジネスロジックの実装

### 学習目的
- 「顧客マスタ」「商品マスタ」「伝票ヘッダ」「伝票明細」といった、複数のマスタとトランザクションデータが関連する複雑なデータモデルを設計・実装する
- 伝票明細行の動的な追加・削除や、合計金額のリアルタイム自動計算など、複雑なUIを実装する（v1とv2で実装の大変さの比較）
- 「見積伝票を確定したら受注伝票を作成する」といった、単なるCRUDを超えたビジネスロジックを実装し、そのロジックをどの層（フロントエンド/バックエンド）に持たせるべきかを考察する

### 作成するバージョン
v1 (従来型) と v2 (モダン型) の両方を作成することで、なぜモダンな構成（特にReact）が複雑な業務UI（動的フォーム）で好まれるのかを、実装コストや保守性の観点から深く比較・理解できます。

## ロードマップ概要

| ステップ | テーマ | 難易度 | v1 | v2 | 主な学習ポイント |
|---------|--------|--------|-----|-----|-----------------|
| 1 | メモ帳（CRUD） | ⭐ | ✅ | ✅ | MVC vs SPA、基本的なCRUD |
| 2 | タスク管理（1対多） | ⭐⭐ | (オプション) | ✅ | リレーション、状態管理 |
| 3 | 伝票処理（複雑） | ⭐⭐⭐ | ✅ | ✅ | 複雑なUIロジック、ビジネスロジック層 |
