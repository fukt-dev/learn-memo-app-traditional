第 1 章

はじめに

1.1 この手順書について

本手順書は、開発未経験者がメモ帳アプリケーションを開発できるように、必要なソフトウェアのインストールと設定方法を 1 つずつ丁寧に 説明します。Docker 方式（推奨）は Docker Desktop のみで最短 10 分で起動可能、ローカル開発方式（従来型）は Java、Maven 等を個別にインストールする方式です。

1.2 環境構築で何をするのか

プログラムを書いて動かすには、以下のような「道具」が必要です。

道具例え実際の役割Docker 方式ローカル方式
Docker仮想環境全ての環境を 1 つにまとめる✅ 必須PostgreSQL のみ
JDK大工道具セットJava プログラムを書いて実行するための基本ツール❌ 不要✅ 必須
IDE作業台プログラムを書きやすくする編集ツール🔧 コード編集時のみ✅ 必須
Maven材料調達係必要なライブラリを自動で集めてくれるツール❌ 不要✅ 必須
PostgreSQL倉庫データ（メモの内容など）を保存する場所❌ 不要🔧 Docker で代用可
Gitバージョン管理プログラムの変更履歴を記録するツール✅ 推奨✅ 必須

1.3 所要時間

🐳 Docker 方式: 初めての方 = 約 10-20 分（Docker Desktop インストール含む）／ 経験者 = 約 5 分。💻 ローカル開発方式: 初めての方 = 約 2-3 時間 ／ 経験者 = 約 30 分-1 時間。

2.2 推奨する方法

🐳 Docker 方式を推奨する場合:

  • ✅ とにかく最短で動かしたい
  • ✅ 他のプロジェクトと環境を分離したい
  • ✅ Docker を学習したい
  • ✅ チーム開発で環境を統一したい

💻 ローカル開発方式を推奨する場合:

  • ✅ Java 開発の基礎から学びたい
  • ✅ IDE 統合デバッグを活用したい
  • ✅ 既に Java 環境が整っている
  • ✅ Docker をインストールできない環境
初学者へのアドバイス: まず Docker 方式で動かしてアプリケーションの動作を理解し、その後ローカル開発方式で Java 開発の詳細を学ぶのがおすすめです。

🐳 Docker 方式（推奨・最短）

3.1 必要なもの

  • Docker Desktop のみ

3.2 インストール手順

Windows

  1. Docker Desktop のダウンロード: Docker Desktop for Windows にアクセスし、"Download for Windows" をクリック
  2. インストール: ダウンロードした Docker Desktop Installer.exe を実行。インストールウィザードに従って進める。WSL 2 のインストールを求められた場合は指示に従う
  3. Docker Desktop の起動: インストール後、Docker Desktop を起動。初回起動時はチュートリアルが表示されるがスキップ可能
  4. インストール確認:
docker --version
docker compose version

macOS

  1. Docker Desktop のダウンロード: Docker Desktop for Mac にアクセスし、Apple シリコン（M1/M2）版または Intel 版を選択してダウンロード
  2. インストール: ダウンロードした .dmg ファイルを開き、Docker.app を Applications フォルダにドラッグ
  3. Docker Desktop の起動: Applications から Docker を起動。メニューバーに Docker アイコンが表示されるまで待つ
  4. インストール確認:
docker --version
docker compose version

3.3 アプリケーションの起動・3.4 日常的な使い方

3.3 アプリケーションの起動

  1. プロジェクトのクローン:
git clone https://github.com/fukt-dev/learn-memo-app-traditional.git
cd learn-memo-app-traditional
  1. 全サービスの起動:
docker compose up --build

初回起動時の流れ:

  • Maven が依存ライブラリをダウンロード（2-3 分）
  • Java アプリケーションをビルド（2-3 分）
  • PostgreSQL コンテナを起動
  • アプリケーションコンテナを起動
  • 起動時に Flyway がデータベースマイグレーションを自動実行（memos テーブルが作成される）
  1. 起動確認: ブラウザで http://localhost:8080/memos にアクセス。ログに以下が表示されていれば成功です:
Successfully applied 1 migration to schema "public"
Started MemoAppApplication in X.XXX seconds
  1. 停止: Ctrl+C で停止。または別ターミナルで docker compose down

3.4 日常的な使い方

# バックグラウンド起動
docker compose up -d

# ログ確認
docker compose logs -f app

# 停止
docker compose down

# データベースも含めて完全リセット
docker compose down -v
これで完了！ 次は「6. プロジェクトのセットアップ」（26 ページ）へ進んでください。

4.2 Java Development Kit (JDK)

4.2.1 JDK とは何か

JDK (Java Development Kit) は、Java プログラムを書いて実行するための基本ツールセットです。

何ができるようになるか:

  • Java のソースコード（.java ファイル）を書ける
  • Java プログラムをコンパイル（翻訳）して実行できる形（.class ファイル）にする
  • 作成したプログラムを実行する

なぜ JDK 17 が必要か:

  • Spring Boot 3.x は Java 17 以上が必要
  • 最新の機能とセキュリティアップデートを利用できる

4.2.2 JDK のインストール（Windows）

方法 1: Oracle JDK（推奨）

  1. ダウンロード: Oracle JDK ダウンロードページ にアクセスし、"Java 17" を選択。"Windows" タブから "x64 Installer" をダウンロード
  2. インストール: ダウンロードした .exe ファイルをダブルクリック。インストーラーの指示に従う。インストール先はデフォルトのまま進む（例: C:\Program Files\Java\jdk-17）
  3. 環境変数の設定: 環境変数とは、Windows がプログラムを見つけるための「住所録」のようなものです。
1. スタートメニューを右クリック → 「システム」をクリック
2. 「システムの詳細設定」をクリック
3. 「環境変数」ボタンをクリック
4. 「システム環境変数」の「新規」をクリック

   変数名: JAVA_HOME
   変数値: C:\Program Files\Java\jdk-17
   （実際のインストール先に合わせて調整）

5. 「システム環境変数」から「Path」を選択して「編集」
6. 「新規」をクリックして以下を追加:
   %JAVA_HOME%\bin

7. すべてのウィンドウで「OK」をクリック

方法 2: Adoptium（無料版、推奨代替）

  1. Adoptium にアクセス
  2. "Temurin 17 (LTS)" を選択
  3. Windows 用のインストーラーをダウンロード
  4. インストーラーを実行（環境変数は自動設定されます）

4.2 JDK（macOS / Linux・インストール確認）

4.2.2 JDK のインストール（macOS）

Homebrew 使用（推奨）

# Homebrewがインストールされていない場合
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# JDK 17をインストール
brew install openjdk@17

# シンボリックリンク作成
sudo ln -sfn /opt/homebrew/opt/openjdk@17/libexec/openjdk.jdk /Library/Java/JavaVirtualMachines/openjdk-17.jdk

# 環境変数設定（~/.zshrc または ~/.bash_profile に追加）
echo 'export JAVA_HOME=/Library/Java/JavaVirtualMachines/openjdk-17.jdk/Contents/Home' >> ~/.zshrc
echo 'export PATH="$JAVA_HOME/bin:$PATH"' >> ~/.zshrc

# 設定を反映
source ~/.zshrc

4.2.2 JDK のインストール（Linux: Ubuntu/Debian）

# パッケージリストを更新
sudo apt update

# JDK 17をインストール
sudo apt install openjdk-17-jdk -y

# 環境変数設定（~/.bashrc に追加）
echo 'export JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64' >> ~/.bashrc
echo 'export PATH="$JAVA_HOME/bin:$PATH"' >> ~/.bashrc

# 設定を反映
source ~/.bashrc

4.3 統合開発環境 (IDE)

4.3.1 IDE とは何か

IDE (Integrated Development Environment) は、プログラムを書くための「超高機能なエディタ」です。

IDE があると便利なこと:

  • ✅ コードを途中まで書くと、続きを自動で提案してくれる
  • ✅ エラーがあると赤線で教えてくれる
  • ✅ クラスやメソッドをクリック 1 つで定義場所に飛べる
  • ✅ デバッグ（バグ探し）が簡単にできる

IDE は次の 2 つからどちらか 1 つを選べば OK（両方入れる必要はありません）。

  • IntelliJ IDEA Community Edition（4.3.2 で説明）
  • Eclipse (Pleiades All in One)（4.3.3 で説明）── 日本語化済み・JDK / Maven / Lombok 設定済みのオールインワン配布版。これを選ぶと 4.2 JDK と 4.4 Maven のインストールが丸ごと不要になります

迷ったら、個別インストールの手間が少ない Eclipse (Pleiades All in One) がおすすめです。

4.3.2 IntelliJ IDEA のインストール

IntelliJ IDEA は、Java 開発で最も人気のある IDE です。すべての OS 共通手順（Community Edition = 無料版）:

  1. ダウンロード: IntelliJ IDEA ダウンロードページ にアクセス。Community Edition（無料） の「Download」をクリックし、OS に合わせたインストーラーをダウンロード
  2. インストール: Windows は .exe ファイルを実行 ／ macOS は .dmg ファイルを開いてアプリケーションフォルダにドラッグ ／ Linux は .tar.gz を展開して bin/idea.sh を実行
  3. 初回起動設定:
1. IntelliJ IDEAを起動
2. 「Do not import settings」を選択（初回のみ）
3. テーマを選択（DarculaまたはLight）
4. プラグインのインストールは後でOK
5. 「Start using IntelliJ IDEA」をクリック

4.3 Eclipse (Pleiades All in One) のインストール

4.3.3 Eclipse (Pleiades All in One) のインストール

Eclipse (Pleiades All in One) は、Eclipse 本体に日本語化・JDK・Maven・Lombok 等をあらかじめ組み込んだ「オールインワン」配布版です。MergeDoc Project という有志コミュニティが配布しています。

ダウンロード手順:

  1. 検索エンジンで「Pleiades」を検索し、MergeDoc Project のサイトを開く
  2. 「Eclipse 2026」→ エディション「Java」→「Windows x64」の「JRE あり (Full Edition)」を選択
  3. ファイル例: pleiades-2026-06-java-win-64bit-jre_20260705.exe（約 1.3GB）
⚠️ 罠: ダウンロードページには大きな「Download Now」ボタン風の広告が表示されます。本物のダウンロードリンクは、ページ中にある ftp.jaist.ac.jp（JAIST ミラー）へのテキストリンクで、通常はクリックすると自動でダウンロードが始まります。

ファイルの正当性確認（推奨）: ダウンロードした exe ファイルが改ざんされていないか確認できます。ページに記載された MD5 値と、以下のコマンドの結果が一致することを確認してください。

Get-FileHash "ダウンロードしたファイル" -Algorithm MD5

インストール（展開）手順

  1. ダウンロードした exe を実行する（自己解凍形式であり、インストーラーではないため、レジストリや環境変数は一切変更されません）
  2. Microsoft Defender SmartScreen の「Windows によって PC が保護されました」という警告が出たら「詳細情報」→「実行」をクリック（コード署名のない自己解凍形式のファイルであるための警告です。MD5 照合済みであれば、安全に続行できます）
  3. 展開先を指定する。C:\pleiades などの C ドライブ直下の浅いパスにすること（デフォルトの推奨設定に従えば OK）
    • ❌ Program Files 配下 → 権限問題でプラグイン追加や自己更新が壊れる
    • ❌ 深いフォルダ・日本語を含むパス → Windows の 260 文字パス制限に引っかかり、展開や起動に失敗する
  4. 展開が終わったら、以下を実行して起動する:
C:\pleiades\2026-06\eclipse\eclipse.exe

4.6 Docker

4.6.1 Docker とは何か

Docker（ドッカー） は、アプリケーションを「コンテナ」という箱に入れて動かす技術です。

コンテナとは: アプリケーションとその動作に必要なものをすべて詰め込んだ「パッケージ」。どの環境（Windows、macOS、Linux）でも同じように動きます。

例え話:

  • 従来: 引っ越しで家具をバラバラに運ぶ（大変！）
  • Docker: すべてをコンテナ（箱）に詰めて運ぶ（簡単！）

PostgreSQL を Docker で動かすメリット:

  • ✅ インストールが簡単（1 コマンドで起動）
  • ✅ 不要になったら簡単に削除できる
  • ✅ 複数バージョンを同時に動かせる
  • ✅ 本番環境と同じ環境を作れる

4.6.2 Docker のインストール

インストール手順は 3.2 インストール手順（4 ページ）と同じです（Docker 方式・ローカル方式共通）。Linux (Ubuntu) の場合:

# 古いバージョンを削除（存在する場合）
sudo apt-get remove docker docker-engine docker.io containerd runc

# 必要なパッケージをインストール
sudo apt-get update
sudo apt-get install ca-certificates curl gnupg lsb-release

# Dockerの公式GPGキーを追加
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# リポジトリ追加
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Docker Engineをインストール
sudo apt-get update
sudo apt-get install docker-ce docker-ce-cli containerd.io docker-compose-plugin

# 現在のユーザーをdockerグループに追加（sudo不要にする）
sudo usermod -aG docker $USER

# ログアウト・ログインして設定を反映

# PostgreSQLコンテナを起動（お試し用。プロジェクトでは compose を使う）

6.2b Boot ダッシュボードでの起動・停止（推奨）

  2. パッケージ・エクスプローラーで src フォルダを直接展開すると main の下が空に見えるのは正常です: src/main/java などソースフォルダに登録されたパスはツリー上部に独立表示（昇格）されるためです

7.1 Java 関連

問題: java: command not found

原因: 環境変数 PATH が正しく設定されていない。

解決方法:

# JAVA_HOMEが設定されているか確認
echo $JAVA_HOME  # macOS/Linux

# 設定されていない場合
export JAVA_HOME=/path/to/jdk-17
export PATH=$JAVA_HOME/bin:$PATH

問題: Unsupported class file major version XX

原因: プロジェクトが Java 17 を要求しているが、古いバージョンの Java を使っている。

解決方法: JDK 17 をインストールして、JAVA_HOME を更新。

7.2 Maven 関連

問題: mvn: command not found

原因: Maven がインストールされていない、または PATH が通っていない。

解決方法: IntelliJ IDEA 内蔵の Maven を使う（IntelliJ IDEA のターミナルから実行）。

問題: 依存関係のダウンロードに失敗

解決方法:

# キャッシュをクリアして再実行
mvn clean install -U

7.3 Docker 関連

問題: Cannot connect to the Docker daemon

原因: Docker が起動していない。

解決方法: Windows/macOS は Docker Desktop を起動 ／ Linux は sudo systemctl start docker。

問題: ポート 5432 が既に使用されている

原因: 別の PostgreSQL インスタンスが既に動いている。

解決方法:

# 既存のPostgreSQLを停止
sudo systemctl stop postgresql  # Linux
brew services stop postgresql   # macOS

# または、別のポートを使う（docker-compose.ymlを編集）
ports:
  - "5433:5432"  # ホスト側を5433に変更

※ モダン版（learn-memo-app-modern）の PostgreSQL コンテナはホスト側 5433 を使います。両方同時に動かす場合はポートが重ならないようにしてください。

7.5 アプリケーション起動関連

7.5 アプリケーション起動関連

問題: Port 8080 is already in use

原因: 別のアプリケーションがポート 8080 を使用している。

解決方法:

# 使用中のプロセスを確認
# Windows
netstat -ano | findstr :8080

# macOS/Linux
lsof -i :8080

# プロセスを停止するか、別のポートを使う（application.ymlを編集）
server:
  port: 8081

補足（Docker 方式でアプリを動かしたことがある場合）: memoapp-spring コンテナが 8080 を使用中のことがあります → docker stop memoapp-spring で停止してからローカル起動してください（docker start memoapp-spring で元に戻せます）。

問題: Flyway migration failed / Migration checksum mismatch

原因: checksum mismatch = 適用済みのマイグレーションファイルが書き換えられた（Flyway はファイル内容のチェックサムを記録しており、コメント変更でも不一致になります）／ その他 = マイグレーション SQL の文法エラーなど。

解決方法:

# 学習環境ではデータベースのリセットが最も確実（データは消えます）
docker compose down -v
docker compose up -d postgres
mvn spring-boot:run

補足: mvn flyway:repair のようなコマンドは flyway-maven-plugin を導入しているプロジェクト専用で、本プロジェクトでは使えません。マイグレーションの状態は flyway_schema_history テーブルを SQL で直接確認できます:

docker compose exec postgres psql -U memoapp_user -d memoapp -c "SELECT version, description, checksum, success FROM flyway_schema_history;"

7.5 アプリケーション起動関連（Eclipse デバッグ）

問題: デバッグ起動すると SilentExitExceptionHandler.exitCurrentThread() で毎回停止する

原因: spring-boot-devtools の再起動機構が意図的に投げる SilentExitException を、Eclipse のデバッガが「キャッチされない例外」として検出して停止するためです。アプリの異常ではありません。

対処 1（簡単）: そのまま F8（再開）を押せば起動が続行します。

対処 2（恒久）: 「ウィンドウ」→「設定」→「Java」→「デバッグ」→「キャッチされない例外で実行を中断」のチェックを外します。ただし本物のキャッチされない例外でもデバッガが止まらなくなる点に注意してください。
